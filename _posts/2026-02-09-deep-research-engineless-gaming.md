## Executive Summary

2026년 1월 29일, Google DeepMind가 **Project Genie**를 AI Ultra 구독자에게 공개했다. Genie 3 월드 모델은 텍스트 프롬프트 하나로 **720p, 24fps의 인터랙티브 3D 환경**을 실시간 생성한다. 같은 주, 스타트업 LinearGame의 **Yoroll.ai**는 "엔진 없는(engine-less) 게임 플랫폼"이라는 새로운 카테고리를 선언했다. Rosebud AI는 "바이브 코딩"으로 텍스트→플레이 가능한 게임 파이프라인을 대중화하고 있고, Microsoft Research는 **MaaG 프레임워크**로 AI 생성 게임의 일관성 문제를 해결하기 시작했다.

이 모든 것이 의미하는 바는 명확하다: **"프롬프트 → 플레이 가능한 게임"의 시대가 열리고 있다.** 이 리서치는 핵심 플레이어들의 기술 수준과 한계를 분석하고, 22개의 단일 HTML 게임을 운영하는 eastsea.monster가 이 변화 앞에서 어떤 포지셔닝을 취해야 하는지 전략적으로 평가한다.

## 1. Google Genie 3: 월드 모델의 새로운 프론티어

### 1.1 기술 개요

Genie 3는 Google DeepMind가 10년 넘게 축적한 시뮬레이션 환경 연구의 결정체다. Genie 1(2024)이 게임 유사 환경 생성의 개념을 입증했고, Genie 2(2024 말)가 시각적 품질을 향상시켰다면, **Genie 3는 "실시간 인터랙션"이라는 결정적 도약**을 달성했다.

| 항목 | Genie 2 (2024) | Genie 3 (2025.08 연구→2026.01 공개) |
|:---:|:---:|:---:|
| **해상도** | 480p 이하 | **720p** |
| **프레임레이트** | ~10 fps | **20–24 fps** |
| **인터랙션** | 제한적 (비디오 생성 위주) | **실시간 WASD 네비게이션** |
| **일관성 지속** | 수십 초 | **수 분 (비주얼 메모리 ~1분)** |
| **프롬프트 이벤트** | 없음 | **세션 중 날씨·객체 변경 가능** |

핵심 기술 돌파구는 **자기회귀적(auto-regressive) 프레임 생성**에 있다. 매 프레임마다 이전 생성된 전체 궤적(trajectory)을 참조하면서도 실시간으로 사용자 입력에 반응해야 한다. 예를 들어 1분 전에 보았던 장소를 재방문하면, 모델은 1분 전 정보를 참조해 일관된 씬을 재구성한다 — 이 모든 것이 초당 24회 이상 발생한다.

### 1.2 Genie 3의 핵심 능력

**물리적 속성 모델링:** 물의 반사, 잔물결, 유체 운동. 동적 그림자와 시간대 변화. 비, 구름, 안개 전환.

**자연 세계 시뮬레이션:** 생태계 생성 — 동물 행동 패턴, 식물의 디테일까지 포함.

**애니메이션과 픽션:** 판타지 시나리오, 표현력 있는 애니메이션 캐릭터 생성.

**역사적·지리적 탐험:** 고대 로마부터 미래 도시까지, 시공간을 초월한 환경 탐색.

**프롬프트 가능 월드 이벤트:** 네비게이션 입력에 더해 텍스트 기반으로 세계를 변경할 수 있다. 날씨 조건 변경, 새 객체 도입, 환경 이벤트 트리거 — 모두 세션 유지 상태에서 가능.

### 1.3 현재의 한계 — 게임 엔진으로 대체 불가능한 이유

DeepMind 스스로가 인정한 Genie 3의 제약사항은 치명적이다:

1. **제한된 액션 공간:** 주로 네비게이션 기반. WASD 이동과 시점 변경은 가능하지만, 복잡한 조작(아이템 사용, 전투 시스템, 인벤토리 관리)은 지원하지 않는다.

2. **멀티에이전트 인터랙션 부재:** 복수의 독립적 에이전트가 공유 환경에서 상호작용하는 것은 아직 연구 과제. 멀티플레이어? 꿈도 꾸지 말아라.

3. **실제 장소의 정확한 재현 불가:** 그럴듯한 환경은 생성하지만, 실재 위치의 정확한 복원은 아니다.

4. **텍스트 렌더링 취약:** 씬 내에서 읽을 수 있는 텍스트 생성은 여전히 약점.

5. **제한된 인터랙션 시간:** 수 시간이 아니라 **수 분** 단위의 세션만 지원. 전통적 게임처럼 세이브/로드, 수십 시간 플레이는 불가능.

6. **게임 로직 부재:** 점수 시스템, 레벨 진행, 승리/패배 조건, 인벤토리, 퀘스트 트래킹 — 이 모든 것이 **존재하지 않는다.** Genie 3는 "게임"이 아니라 "탐색 가능한 비디오"에 가깝다.

Reddit의 r/gamedev 커뮤니티 반응이 이를 정확히 요약한다:

> "Genie 2와 3 사이 1년이 지났지만, 여전히 persistence(영속성), logic(로직), 실제 gameplay가 없다."

### 1.4 접근성과 비용

현재 Project Genie는 **미국 한정, Google AI Ultra 구독자**($24.99/월)에게만 제공되는 실험적 프로토타입이다. 국제 가용성은 미발표. 이것은 아직 제품이 아니라 **기술 데모**다.

## 2. Yoroll.ai: "엔진 없는" 게임의 상업화를 꿈꾸다

### 2.1 회사 개요

Yoroll.ai는 실리콘밸리-싱가포르 기반 스타트업 **LinearGame**이 개발한 AI 네이티브 인터랙티브 비디오 플랫폼이다. 2025년 12월 프리시드 라운드(수백만 달러)를 확보했으며, "영화를 연출하듯 게임을 만든다"는 비전을 내세운다.

### 2.2 기술 아키텍처 — 3계층 시스템

Yoroll.ai가 제시한 **삼중 계층 아키텍처(Three-Layer Architecture)**는 AI 생성 게임의 핵심 문제인 "환각(hallucination)"을 해결하려는 의미 있는 시도다:

| 계층 | 역할 | 기술 |
|:---:|------|------|
| **표현 계층 (Expression Layer)** | 렌더링 엔진 역할. 시각적 출력과 즉각적 물리 피드백 | Genie 3 또는 자체 Roll-01 모델 |
| **판단 계층 (Judgment Layer)** | 실시간 AI 심판. VLM이 비디오 스트림 모니터링 | Vision-Language Model (VLM) |
| **상태 계층 (State Layer)** | 인벤토리, HP, 분기 플롯 등 결정적 데이터 저장 | 전통적 결정론적 데이터베이스 |

이것이 중요한 이유: **세계 모델이 시각적으로 환각해도 플레이어의 진행 상태는 보존된다.** VLM 관찰자가 "플레이어가 공격을 회피했는가?", "퀘스트 아이템을 획득했는가?"를 픽셀에서 판독하고, 상태 계층에 반영한다.

### 2.3 타겟 시장과 비즈니스 모델

Yoroll.ai는 AAA 게임 대체가 아니라 **인터랙티브 시네마틱 경험**을 노린다. Black Mirror: Bandersnatch 같은 분기형 내러티브를, AI 비용으로 제작하겠다는 것이다.

LinearGame의 주장에 따르면:
- 제작 비용을 전통 인터랙티브 필름의 **1/100**로 절감
- 수십 명, 수년 걸리던 프로젝트를 **1~3명이 1달**에 완성
- TikTok 크리에이터와 인디 영화감독이 게임 디자이너가 되는 "로블록스 모멘트" 창출

### 2.4 현실적 평가

낙관적 비전이지만 현실은 냉정하다:
- 아직 **공개 데모가 없다** — 발표된 것은 프리시드 투자와 비전 스테이트먼트
- 자체 Roll-01 모델의 실제 품질은 미검증
- VLM 기반 판단 계층의 정확도와 지연시간 문제는 해결 과제
- "인터랙티브 필름"은 게임 시장의 아주 작은 니치(niche)

## 3. Rosebud AI: 바이브 코딩으로 텍스트→게임

### 3.1 접근 방식

Rosebud AI는 Genie 3나 Yoroll과는 근본적으로 다른 접근법을 취한다. 월드 모델(비디오 생성)이 아니라, **LLM 기반 코드 생성**으로 실제 플레이 가능한 게임을 만든다.

핵심 파이프라인:
1. **자연어 설명** → AI가 게임 코드, 스프라이트, 배경 에셋 자동 생성
2. 브라우저에서 즉시 **플레이 가능한 2D/3D 게임** 출력
3. 코드가 실제 존재하므로 **수정·확장이 가능**
4. itch.io 퍼블리시 지원

"바이브 코딩(Vibe Coding)"이라는 브랜딩을 사용하며, 코딩 경험이 없는 사용자도 게임을 만들 수 있다고 홍보한다.

### 3.2 Genie 3와의 차이점

| 비교 항목 | Genie 3 (월드 모델) | Rosebud AI (코드 생성) |
|:---:|:---:|:---:|
| **출력물** | 탐색 가능한 비디오 스트림 | 실제 실행 가능한 게임 코드 |
| **게임 로직** | 없음 | 있음 (AI가 코드로 구현) |
| **에셋** | AI가 매 프레임 생성 | AI가 스프라이트/배경 생성 |
| **수정 가능성** | 프롬프트로만 변경 | 코드 직접 수정 가능 |
| **세이브/로드** | 불가 | 구현 가능 |
| **배포** | 불가 (Google 서버 의존) | 웹 배포 가능 |
| **품질** | 포토리얼리스틱 비주얼 | 2D 픽셀아트~기본 3D |

Rosebud는 현재 가장 **실용적인** AI 게임 생성 도구다. 하지만 출력 품질은 프로토타입~인디 수준이며, 복잡한 게임 메커닉을 안정적으로 생성하는 데는 한계가 있다.

### 3.3 Genie 3 + Rosebud 시너지

흥미롭게도 Rosebud는 이미 **"Genie 3 Planner"** 게임 템플릿을 출시했다. 사용자가 게임적 단계(장르 선택 → 바이옴/분위기 → 액션/인터랙션 → 퀘스트)를 거치면, Genie 3에 넣을 수 있는 프롬프트 패키지를 자동 생성한다. 두 도구의 결합이 시작되고 있다.

## 4. 기타 플레이어와 생태계 지도

### 4.1 에셋 생성 특화 도구

| 플랫폼 | 핵심 기능 | 한계 |
|--------|----------|------|
| **Scenario.com** | AI 기반 게임 아트 생성, 스타일 파인튜닝, 배치 이미지 생성 | 에셋만 생성. 게임 로직/실행 불가 |
| **Leonardo AI** | 범용 이미지 생성, 게임 에셋 최적화 | Scenario보다 범용적이지만 게임 특화 약함 |
| **Layer.ai** | 3D 레이어 생성 (지형, 식생 등) | 파이프라인 보조 도구 |

### 4.2 게임 디자인/리서치 도구

| 플랫폼 | 핵심 기능 | 한계 |
|--------|----------|------|
| **Ludo.ai** | 시장 리서치, 트렌드 분석, 게임 아이디어 생성, 3D 에셋 생성 | 게임 자체를 만들지는 않음 |
| **Charisma.ai** | AI 기반 동적 내러티브/대화 생성 | 스토리 전용, 게임플레이 로직 없음 |
| **InWorld AI** | 포토리얼 캐릭터 생성, 지능형 NPC | NPC 특화, 전체 게임 생성 불가 |

### 4.3 풀스택 게임 생성 시도

| 플랫폼 | 핵심 기능 | 한계 |
|--------|----------|------|
| **Exists.ai** | 텍스트→3D 월드+팀 데스매치 | Reddit 평가: "글로리파이드 레벨 에디터". 클론 느낌. |
| **Rosebud AI** | 텍스트→실행 가능한 2D/3D 게임 | 복잡한 게임 로직 한계, 인디 수준 비주얼 |
| **Yoroll.ai** | 엔진리스 인터랙티브 시네마 | 아직 미출시. 비전 단계 |

### 4.4 Microsoft MaaG 프레임워크

Microsoft Research가 발표한 **Model as a Game(MaaG)**은 학술적으로 매우 중요한 진전이다. AI 생성 게임의 두 가지 핵심 일관성 문제를 정면으로 다룬다:

- **수치적 일관성(Numerical Consistency):** 점수가 정확하게 증감하는가? LogicNet이 게임 이벤트 발생 여부를 판단하고, 외부에서 점수를 계산해 모델에 재주입.
- **공간적 일관성(Spatial Consistency):** External Map이 탐색된 모든 씬을 영구 저장. 프레임 생성 전 이 맵을 참조해 시각적 연속성 유지.

추론 지연은 약 **0.015초**로, 게임플레이 유동성을 유지한다. 하지만 아직 Traveler, Pong, Pac-Man 같은 **극도로 단순한 2D 게임**에서만 검증되었다.

## 5. 생태계 성숙도 매트릭스

현재 "AI → 게임" 생태계를 4단계로 정리하면:

```
[1단계: 에셋 보조]     ← Scenario, Leonardo, Layer, Ludo
  AI가 텍스처, 스프라이트, 3D 모델 생성
  게임 자체는 인간이 전통 엔진으로 제작
  ✅ 이미 실용 수준

[2단계: 코드 생성]     ← Rosebud AI, GitHub Copilot
  AI가 게임 코드 + 에셋을 함께 생성
  결정론적 게임 로직 존재
  ⚠️ 단순 게임은 OK, 복잡한 게임은 불안정

[3단계: 월드 모델]     ← Genie 3, Yoroll (비전)
  AI가 프레임 단위로 세계를 실시간 생성
  인터랙티브하지만 게임 "로직"은 부재
  ⚠️ 기술 데모 수준. 상업 게임으로는 부족

[4단계: 완전 자율 게임 생성]  ← 아직 없음
  프롬프트 하나로 세이브/로드, 멀티플레이어,
  수십 시간 플레이 가능한 완성 게임
  ❌ 3년 이상 소요 예상
```

## 6. eastsea.monster 영향 분석

### 6.1 현재 포지션 재확인

eastsea.monster는 22개의 단일 HTML 파일 게임을 호스팅하는 캐주얼 게임 포털이다. 각 게임은:
- **단일 HTML 파일**로 자기 완결적 (CDN 최적, 오프라인 가능)
- 결정론적 게임 로직 내장 (점수, 레벨, 승/패)
- 모바일 반응형
- Rust(WASM) + Godot 파이프라인으로 전환 중

### 6.2 AI 생성 게임과의 차별화 요소

| 차별화 포인트 | eastsea.monster | AI 생성 게임 (Genie 3 등) |
|:---:|:---:|:---:|
| **게임 로직** | ✅ 완벽한 점수/레벨/진행 시스템 | ❌ 부재 또는 불안정 |
| **일관성** | ✅ 결정론적 — 항상 동일 동작 | ⚠️ 확률적 — 환각 가능 |
| **세션 길이** | ✅ 무제한 | ❌ 수 분 한정 |
| **오프라인 지원** | ✅ 단일 HTML | ❌ 클라우드 의존 |
| **배포 경량성** | ✅ < 1MB per game | ❌ GPU 서버 필요 |
| **멀티플레이어** | ⚠️ 제한적 | ❌ 미지원 |
| **비주얼 품질** | ⚠️ 2D 픽셀아트~기본 3D | ✅ 포토리얼리스틱 |
| **콘텐츠 다양성** | ⚠️ 수동 제작 (22개 고정) | ✅ 무한 변형 가능 |
| **제작 비용** | ⚠️ 게임당 수일~수주 | ✅ 수 분~수 시간 |

핵심 인사이트: **"게임"이라는 단어의 정의가 분기하고 있다.**

Genie 3 류의 월드 모델이 만드는 것은 엄밀히 "게임"이 아니라 **"인터랙티브 경험"**이다. 게임의 본질적 요소 — 목표, 규칙, 도전, 피드백 루프 — 가 구조적으로 내장되어 있지 않다. 반면 eastsea.monster의 게임들은 비록 소규모지만 **완전한 게임 루프**를 갖추고 있다.

### 6.3 위협 시나리오 분석

**시나리오 A: Rosebud형 도구가 "한판만 더" 게임을 즉석 생성 (위험도: 🟡 중)**

Rosebud AI가 이미 할 수 있는 것: "테트리스 클론 만들어줘" → 30초 안에 플레이 가능한 게임 생성. 이것은 eastsea.monster의 일부 단순 게임과 직접 경쟁한다.

하지만 현실적 사용 패턴을 보면:
- Rosebud로 생성한 게임은 **일회성 신기함**에 가깝다
- "만들어보기"의 재미이지, "반복 플레이"의 재미가 아니다
- 밸런싱, 난이도 곡선, 중독성 루프 — 이런 것은 AI가 자동으로 달성하지 못한다

**시나리오 B: Genie 3가 진화하여 게임 로직을 내장 (위험도: 🟢 낮음, 단기)**

MaaG 같은 프레임워크가 Genie 3에 통합되어 점수, 인벤토리, 퀘스트가 작동하는 세계가 생성되는 시나리오. 기술적으로 가능하지만, 현재 MaaG조차 Pong 수준에서 검증 단계. **3년 이상** 소요 예상.

**시나리오 C: Yoroll형 하이브리드가 주류가 됨 (위험도: 🟡 중, 중기)**

Yoroll의 3계층 아키텍처(표현+판단+상태)가 성숙하면, "비주얼은 AI, 로직은 결정론적"인 하이브리드 게임이 등장한다. 이것이 가장 현실적인 위협이지만, Yoroll 자체가 아직 미출시이며 인터랙티브 필름이라는 니치를 타겟.

### 6.4 기회 시나리오 분석

**기회 1: AI를 에셋 파이프라인에 통합 (즉시 실행 가능)**

- Scenario/Leonardo AI로 게임 에셋(스프라이트, 배경, UI) 자동 생성
- 게임당 에셋 제작 시간을 **수일 → 수 시간**으로 단축
- 이미 Kenney CC0 + AI 생성 에셋을 사용하고 있으므로, 파이프라인 강화

**기회 2: Rosebud/AI 코딩을 프로토타이핑에 활용 (3개월 내)**

- Rosebud 또는 Claude/GPT 기반 "바이브 코딩"으로 게임 컨셉 빠르게 프로토타이핑
- 프로토타입 → 검증 → Rust(WASM)/Godot로 본격 구현
- 22개 → 50개 이상으로 포트폴리오 확장 가속

**기회 3: "큐레이티드 AI 게임 포털" 피벗 (6개월~1년)**

- AI가 생성한 게임을 **큐레이션·밸런싱·품질보증**하여 제공하는 플랫폼
- AI 게임 생성이 보편화될수록, "수많은 AI 게임 중 진짜 재미있는 것"을 가려주는 가치 증가
- "AI 슬롭 게임" 홍수 속 큐레이터 역할

**기회 4: 하이브리드 모델 — AI 월드 + 결정론적 로직 (1~2년)**

- Genie 3 또는 후속 월드 모델을 비주얼 레이어로 사용
- eastsea.monster의 게임 로직 엔진을 상태 관리 레이어로 유지
- Yoroll의 아이디어를 자체적으로 경량 구현

### 6.5 타임라인 평가

| 시점 | AI 게임 생성 수준 | eastsea.monster 영향도 |
|:---:|:---:|:---:|
| **현재~6개월** | 기술 데모 수준. Genie 3 = 탐색 가능한 비디오. Rosebud = 단순 게임만 안정적 생성. | **영향 미미.** 경량 HTML 게임의 가치는 유지됨 |
| **6개월~1년** | 코드 생성 AI 품질 향상. Rosebud 2세대, Claude/GPT 게임 코딩 능력 상승. 중간 복잡도 게임 생성 가능. | **주의 필요.** 단순 캐주얼 게임 영역에서 경쟁 시작 |
| **1~2년** | 월드 모델 + 로직 레이어 통합 시작. Yoroll류 상업 플랫폼 출시. | **적극 대응 필요.** 차별화 전략 필수 |
| **2~3년** | 프롬프트→중간 복잡도 완성 게임이 보편화. 하지만 AAA·복잡한 시스템 게임은 여전히 인간 영역. | **포지셔닝 재정의** 시점 |

## 7. 전략 제안

### 7.1 결론: "하이브리드" 전략 채택

**현행 유지도, 전면 피벗도 아닌 "하이브리드" 전략**이 최적이다.

### 7.2 즉시 실행 (0~3개월)

1. **에셋 파이프라인 AI 통합**
   - Scenario.com 또는 MLX 기반 로컬 이미지 생성으로 게임 에셋 가속
   - 게임당 아트 에셋 제작 시간 70% 절감 목표

2. **포트폴리오 급확장**
   - AI 프로토타이핑 → 수동 밸런싱 → Godot/WASM 빌드 파이프라인
   - 22개 → 6개월 내 40개 목표
   - "양"이 곧 방어벽 — AI가 만드는 것은 쉽지만, 밸런싱된 40개 게임 포털은 AI가 아직 못 만든다

3. **게임 로직 라이브러리 구축**
   - 재사용 가능한 점수/레벨/세이브 시스템 모듈화
   - Rust WASM 컴포넌트로 패키지화
   - 이것이 AI 시대의 "해자(moat)"가 된다

### 7.3 중기 실행 (3~12개월)

4. **"AI 어시스트 게임 팩토리" 구축**
   - 사용자가 프롬프트로 게임 아이디어 제출
   - AI가 프로토타입 생성 → 인간이 밸런싱/QA → 포털에 게시
   - 크리에이터 에코시스템의 시작

5. **기술 모니터링 체계**
   - Genie 4(예상), Rosebud 2.0, Yoroll 정식 출시를 매월 추적
   - MaaG 후속 논문, 오픈소스 월드 모델 프로젝트 모니터링
   - 임계점(tipping point) 시그널: "Rosebud로 만든 게임이 itch.io 인기 순위 Top 50에 진입"

### 7.4 장기 옵션 (1~3년)

6. **하이브리드 렌더링 실험**
   - 오픈소스 월드 모델이 충분히 성숙하면, 배경 비주얼을 AI 생성으로 전환
   - 게임 로직은 Rust WASM으로 유지 (결정론적)
   - "포토리얼 비주얼 + 완벽한 게임 로직"의 결합

7. **큐레이션 플랫폼 진화**
   - AI 게임 생성이 보편화되면, "만들기"가 아니라 "발견/추천"이 가치
   - eastsea.monster가 "AI 게임의 App Store" 역할

## 8. 맺음말: 엔진의 죽음인가, 엔진의 진화인가

"엔진 없는 게임"이라는 표현은 과장이다. 실제로 일어나고 있는 것은 **엔진의 추상화 레벨이 한 단계 올라가는 것**이다.

- 1990년대: 하드웨어 직접 제어 → **게임 엔진** 등장
- 2010년대: 엔진의 대중화 (Unity, Unreal) → **비주얼 스크립팅** 등장
- 2020년대 중반: AI가 코드를 생성 → **프롬프트 프로그래밍** 등장
- 2026년~: 월드 모델이 프레임을 생성 → **"프롬프트 렌더링"** 등장

각 단계마다 "전통 방식의 죽음"이 선언되었지만, 실제로는 **도구의 계층이 하나 추가**되었을 뿐이다. C++로 게임을 짜는 사람도, Unity로 드래그앤드롭하는 사람도, Rosebud로 바이브코딩하는 사람도 — 모두 공존한다.

eastsea.monster의 가장 큰 자산은 **"작동하는 게임"**을 보유하고 있다는 것이다. AI가 아무리 아름다운 세계를 생성해도, "게임 오버 후 한판 더"를 누르게 만드는 중독성 루프는 여전히 인간의 디자인 영역이다. 이 핵심 역량을 유지하면서, AI가 제공하는 생산성 도구를 적극 수용하는 것 — 이것이 2026년 eastsea.monster의 최적 전략이다.
