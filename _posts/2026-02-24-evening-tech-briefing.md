- **GitHub Actions, workflow dispatch API가 run ID/URL 직접 반환 지원** (GitHub Changelog)
  GitHub는 workflow dispatch API에 `return_run_details` 옵션을 도입해 호출 직후 실행 워크플로우 메타데이터를 받을 수 있게 했습니다.
  기존 204 No Content 응답과 달리 이제 200 OK 응답으로 workflow ID·API URL·워크플로우 URL을 받아 추적이 가능하고, GitHub CLI v2.87.0에서도 `gh workflow run` 후 run URL 출력이 지원됩니다.
  임팩트는 CI 자동화에서 폴링·매핑 로직 복잡도를 크게 줄일 수 있다는 점이며, 운영팀은 실행 추적 코드와 실패 알림 파이프라인을 더 단순한 구조로 리팩터링할 수 있습니다.
  → 링크: https://github.blog/changelog/2026-02-19-workflow-dispatch-api-now-returns-run-ids

- **Qiita 트렌드: Docker 이름있는 볼륨이 의존성 갱신을 가리는 실전 장애 사례 확산** (Qiita)
  Qiita 인기글에서는 `docker compose up -d --build` 이후에도 신규 패키지가 반영되지 않는 문제를 이름있는 볼륨(node_modules) 동작으로 해부했습니다.
  핵심은 볼륨이 비어 있을 때만 이미지 내용을 초기 복사하고, 이후에는 기존 볼륨이 우선되어 브랜치 전환 시 `ERR_MODULE_NOT_FOUND`가 발생할 수 있다는 점이며, 볼륨 선택 삭제·기동 시 install 재실행 같은 대응안을 제시합니다.
  시사점은 Dev 환경 장애의 상당수가 코드가 아니라 스토리지 라이프사이클 불일치에서 나온다는 점이며, 팀 템플릿에 “의존성 변경 시 볼륨 재생성 규칙”을 명시해야 반복 장애를 줄일 수 있습니다.
  → 링크: https://qiita.com/kuma_3838/items/a116aa00af0884675b25

- **Qiita 트렌드: Envoy AI Gateway 로컬 K8s 실험으로 멀티 LLM 라우팅 실무 검증** (Qiita)
  또 다른 트렌드 글은 kind 기반 로컬 Kubernetes에 Envoy AI Gateway를 올리고, 두 개의 테스트 LLM 업스트림을 헤더 조건으로 분기시키는 과정을 단계별로 공개했습니다.

  임팩트는 멀티모델 운영이 이론이 아니라 인프라 레벨에서 빠르게 프로토타이핑 가능한 단계에 들어섰다는 점이며, AI 서비스 팀은 모델 교체 실험을 앱 코드가 아닌 게이트웨이 정책에서 먼저 검증하는 방식이 비용 효율적입니다.
