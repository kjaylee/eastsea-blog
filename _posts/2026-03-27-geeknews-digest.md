### 1. 장기 실행 애플리케이션 개발을 위한 하네스 설계 (33pts)
**[1. 장기 실행 애플리케이션 개발을 위한 하네스 설계]** (33pts)
**요약**: Anthropic은 단기 요청 처리와 단절 없는 장기 실행 작업을 동일한 운영 체계로 다루기 위해 "하네스" 개념을 제안했다. 짧은 인터랙션 중심의 기존 에이전트 설계가 장시간 반복 작업에서 맥락 손실, 재개 비용, 품질 저하를 자주 일으킨다는 문제를 먼저 집어냈다. 제안된 모델은 생성자(또는 실행기), 평가기, 감시 루프를 분리해 긴 작업을 단계적으로 나누고, 각 단계마다 검증 신호를 남기는 방식이다. 핵심은 작업을 API 호출 단위로만 분해하는 것이 아니라, 실행 상태와 정책을 저장 가능한 형태로 감싸는 것이다. 결국 “계속 잘 돌아가는 채팅 봇”이 아니라 “실패를 줄이고 재개 가능한 장기 실행 시스템”에 가깝다. 긴 세션에서의 규칙 위반, 품질 편차를 줄이기 위해 재현 가능한 체크포인트와 반복 제어가 들어간 점이 가장 중요하다.
**기술적 배경**: 이 글은 최근 오랜 맥락을 요구하는 LLM 워크플로(백로그 정리, 대규모 리팩터, 배포 자동화)가 단발성 프롬프트 프레임을 벗어나고 있음을 반영한다. 기존은 대체로 단일 루프 에이전트 방식이었고, 에러 복구/회귀 추적에 약했다. 하네스 방식은 상태 저장, 감사 가능성, 작업 분할을 통해 오케스트레이션 레이어를 강화해 기존 에이전트 프레임과 차별화한다. 기존 오픈소스 에이전트 도구들이 CLI 중심·세션 중심으로 머문 반면, 이 방식은 제품 수준 운영을 고려한 내구성 설계에 가깝다.
**영향 분석**: 개발자에게는 "한 번 실행해 끝"형 코딩이 아니라 “실행 과정이 추적되는 자동화”로 전환할 필요를 뜻한다. 스타트업은 장기 자동화 스크립트, 데이터 파이프라인 정비, 배포 사후 검증 자동화를 하네스형으로 바꾸면 운영 안정성이 급상승한다. 인디 빌더는 규모가 작아도 장기 실행 작업(빌드 캐시 갱신, 스케줄링, 린트 보정 루프)에 실패 회복이 쉬운 구조를 얻는다. OpenClaw/eastsea 파이프라인에서는 반복되는 AI 오케스트레이션 작업이 “중단 후 이어붙이기”에 강해질 수 있다.
**Master 액션 포인트**: 1) OpenClaw의 게임빌드/배포 파이프라인에서 상태 점검 토큰(점검 스냅샷)을 넣어 장기 작업이 실패해도 재개 가능하게 구조화한다. 2) `posts` 자동 발행 크론이나 블로그 수집 파이프라인에 작업별 checkpoint 로그(요약+리스크 상태) 포맷을 도입해 회귀 추적성을 확보한다.
- 원문: https://www.anthropic.com/engineering/harness-design-long-running-apps

### 2. understudy - 시연으로 배우는 로컬 데스크톱 에이전트 (2pts)
**[2. understudy - 시연으로 배우는 로컬 데스크톱 에이전트]** (2pts)
**요약**: Understudy는 단일 애플리케이션에서 브라우저, 데스크톱, CLI, 메시지 채널까지 같은 런타임으로 다루려는 로컬형 에이전트이다. 사용자가 GUI 작업을 한 번 보여주면 의도를 학습해 반복 실행하고, 같은 작업을 특정 좌표에 결합하지 않고 경로 자체로 정리하려는 시도를 둔다. 오픈 소스와 개인 API 키 기반 모델 사용으로 비용 제어나 데이터 소유권을 유지할 수 있고, 구독형 전용 제품과 차별된다. 저장·편집 루프, 멀티 채널 처리, 데모 기반 워크플로라는 조합은 단순 자동화와는 다른 ‘업무 동작을 계승 가능한 스킬’로의 진화이다. 실사용 측면에서 화면 기반 학습이 포함되기 때문에 단순 텍스트 에이전트에서 놓치기 쉬운 GUI 자동화까지 확장된다. README 수준에서도 1~2단계의 성숙도로 운영되는 “데모→정제→실행” 순서가 강조되어 장기 안정화 계획이 보인다.
**기술적 배경**: 최근 에이전트는 텍스트·파일·브라우저 API 위주였지만, 실제 실무는 데스크톱 앱, 알림, 채널 통합이 섞여 있다. Understudy의 차별점은 채널 통합과 화면 기반 지시를 하나의 로컬 런타임에 묶어 제공한다는 점이다. 구독형 SaaS보다 데이터가 외부로 나가지 않으면서 로컬에서 동작하므로 보안 요구가 높은 팀이 선호할 수 있다. 기존 오픈AI 기반 커맨드형 툴과 달리 ‘시연 기반 학습’은 반복성이 높은 작업에서 초기 온보딩 비용을 낮춘다.
**영향 분석**: 개발자 입장에서는 반복 수작업(빌드 결과 정리, 리소스 업로드, 경로 이동 등)을 텍스트 프롬프트만으로는 자동화가 어려운 구간까지 확장할 수 있다. 스타트업은 데모형 자동화를 통해 비개발 멤버도 루틴을 빠르게 정착할 수 있고, 운영팀은 반복 보고 루틴을 줄일 수 있다. 인디 빌더는 비용 제어된 로컬 스택에서 고비용 API 자동화보다 확장성 좋은 선택을 가질 수 있다.
**Master 액션 포인트**: 1) eastsea 게임빌드 파이프라인에서 자주 반복되는 GUI 동작(콘텐츠 업로드, 버전 검토)을 데모 기반 스킬로 전환 테스트한다. 2) 메시지 채널(텔레그램/디스코드)에서 들어오는 요청을 Understudy가 받아 정형화된 배치로 처리하도록 브릿지를 실험한다.
- 원문: https://github.com/understudy-ai/understudy

### 3. Upright - Playwright와 Prometheus 메트릭을 활용한 합성 모니터링 엔진 (2pts)
**[3. Upright - Playwright와 Prometheus 메트릭을 활용한 합성 모니터링 엔진]** (2pts)
**요약**: Upright는 다중 지리 위치에서 HTTP/SMTP/트레이스라우트/브라우저(Playwright) Probe를 주기적으로 실행해 상태를 기록하는 합성 모니터링 시스템이다. 실서비스 장애 탐지는 단일 체크포인트가 아닌 지역별 가용성 맥락을 반영해야 한다는 점을 전제로 설계됐다. 수집된 결과를 Prometheus 메트릭으로 노출하고 AlertManager와 연동해 알림 체계를 외부 관측 스택에 붙이도록 구성한다. 설치는 Rails 엔진 기반으로, 사이트별 서브도메인 라우팅과 작업 스케줄링(Solid Queue)까지 묶어서 제공한다. 영상·로그 기록, 인증/트래픽 설정이 함께 고려되어 운영 환경 배포를 염두에 둔 시스템이다. 문서상 VM 최소 사양이 비교적 가볍게 제시되어 소규모 팀도 실전 적용이 가능하다.
**기술적 배경**: 과거 합성 모니터링은 체크만 제공하거나 모니터링 전문 플랫폼에 의존하는 경우가 많았다. Upright는 Playwright 기반 브라우저 플로우 검증까지 결합해 단순 ping보다 사용자 관점 장애를 포착한다. Prometheus/OTLP/AlertManager와의 자연스러운 결합은 기존 클라우드 인프라와 쉽게 맞물려 대체 비용을 낮춘다. 대형 서비스 대비 소규모 오픈 소스 팀도 운영형 관측 체계를 갖추기 위한 ‘저비용 템플릿’ 역할을 한다.
**영향 분석**: 개발자에게는 실제 사용자가 보게 되는 장애 경보를 더 빨리 감지해 디버깅 루프가 짧아진다. 스타트업은 모니터링 아웃소싱 없이 내부 인력으로 SLA 대응 체계를 갖출 수 있다. 인디 빌더에게는 게임 빌드 배포 후 smoke test를 지역별/브라우저별로 자동 검증하는 데 직접적인 효익이 있다. eastsea의 웹앱, 배포 페이지, 스크립트 인프라를 운영한다면 배포 직후 모니터링 결합 시 장애 발견 시간이 크게 줄 수 있다.
**Master 액션 포인트**: 1) eastsea.github.io 정적 배포 후 `/health` + 핵심 페이지를 Playwright Probe로 30초/15분 주기 이중 검사한다. 2) 현재 CI에 Prometheus 연동 webhook을 붙여 알림을 OpenClaw 알림 루틴과 연계한다.
- 원문: https://github.com/basecamp/upright

### 4. gitagent - AI 에이전트 정의 및 관리를 위한 Git 기반 표준 (21pts)
**[4. gitagent - AI 에이전트 정의 및 관리를 위한 Git 기반 표준]** (21pts)
**요약**: gitagent는 AI 에이전트 설정을 프레임워크에 종속되지 않게 레포지토리 안에 표준화하려는 시도다. 핵심은 `agent.yaml`, `SOUL.md`, 규칙/역할 정의를 통해 에이전트를 “코드의 한 산출물”로 관리한다는 점이다. 저장소를 클론하면 곧바로 에이전트로 해석 가능한 구조를 제공해 버전 관리, 브랜칭, PR 리뷰, 감사 추적까지 자연스럽게 연동된다. 단일 프레임워크 전용 커스텀 스크립트가 아니라, 여러 런타임으로 export 가능한 구조가 강조된다. compliance 관점에서는 역할 분리, 승인 플로우(SOD), 휴먼-in-the-loop 정책을 스펙에 넣을 수 있어 규제 환경 대응이 가능하다. 실전은 작은 manifest 파일 설계보다 장기 운영 규칙의 정교성이 성패를 가른다는 메시지가 강하다.
**기술적 배경**: 현재 에이전트 도구별 설정이 각기 다르다 보니 팀 차원의 공유와 이관이 어렵다. gitagent는 이를 Git-native 포맷으로 통일해 기존 GitOps 사고방식에 맞춘다. 기존 방식은 에이전트 프롬프트·코드·컴플라이언스가 흩어져 관리되기 쉬웠지만, gitagent는 이를 단일 트리로 정리한다. CLI/CI 통합이 잘된다는 점에서 “기획-개발-배포” 전 주기의 문서형 표준으로 적합하다.
**영향 분석**: 개발자에게는 에이전트 버전 관리가 쉬워지고, 팀 간 handoff 실패를 줄일 수 있다. 스타트업은 규제 대응(금융/보안)에서 책임 분리와 승인 경로를 깔끔하게 구현할 수 있다. 인디 빌더는 실험적 아이디어를 문서+검증 단계로 붙여두어, “좋은 프롬프트”가 아니라 “재현 가능한 에이전트”로 운영 가능하다. OpenClaw에선 오케스트레이션 루틴 자체를 에이전트화할 때 운영 안정성을 크게 끌어올릴 수 있다.
**Master 액션 포인트**: 1) Eastsea의 크론/발행 에이전트 세트를 `agent.yaml`+규칙 문서로 정식 표준화해 PR 기반으로 관리한다. 2) 서브에이전트 역할 분리(기획/검토/배포) 정책을 도입해 단일 실패가 전부를 막지 않게 한다.
- 원문: https://github.com/open-gitagent/gitagent

### 5. Claude Code로 20년 전 상용 게임을 거의 고치지 않고 브라우저로 이식하기까지 (38pts)
**[5. Claude Code로 20년 전 상용 게임을 거의 고치지 않고 브라우저로 이식하기까지]** (38pts)
**요약**: 기존 Windows 전용 데스크톱 게임인 GunZ: The Duel을 거의 원본 소스를 유지한 채 WebAssembly+WebGL로 브라우저에 포팅한 사례다. 저자는 AI(Claude Code 포함)를 활용해 빌드/호환 작업의 비중을 크게 줄였고, 핵심은 전체 이식이 아니라 기존 아키텍처를 보존한 상태에서 플랫폼 계층만 바꾼 점이다. 게임 실행 흐름, 입력 체계, 렌더링 경로를 유지하면서도 브라우저 접근성을 크게 개선해 설치 없이 즉시 플레이가 가능해졌다. 텍스트로는 단순 포팅처럼 보여도, 실제로는 빌드 파이프라인·자산 번들링·브라우저 호환성·네트워크 연동을 함께 건드린 고난이도 작업이다. 링크에는 이미 플레이 가능한 데모 URL이 제공되어 실제 UX 차이를 쉽게 검증할 수 있다. 과거 레거시 엔터테인먼트 자산의 회수 가치가 얼마나 높은지 보여주는 좋은 사례다.
**기술적 배경**: 레거시 게임/에뮬레이션 영역에서 클라이언트 종속성이 큰 프로젝트는 장비 제약 때문에 사용자층이 제한됐다. WebAssembly는 이런 구형 자산의 플랫폼 확장에 핵심 수단이지만, 실전에서는 타이밍, 입출력, 사운드 처리, 그래픽 API 차이를 정교하게 맞춰야 한다. 이번 사례는 AI가 반복 변환 작업을 가속하지만, 여전히 도메인 판단(무엇을 유지할지, 어디서 포기할지)을 사람이 결정한다는 점을 보여준다. 기존엔 포팅이 “전면 리메이크”로 흐르기 쉬웠다는 점 대비, 소스 보존형 접근이 비용 대비 효과가 높다는 점이 차별점이다.

### 9. 속도를 늦춰야 빨라진다 (14pts)

**기술적 배경**: 다중 에이전트 생태계의 가장 큰 운영 문제는 스킬과 설정 분산이다. Chops는 파일 시스템 수준에서 이를 정리하는 UI형 레이어를 제공한다는 점이 차별점이다. 기존에는 IDE별로 분산된 rule을 직접 찾아 편집해야 했지만, 이 앱은 중앙 탐색을 지원해 관리 오버헤드를 줄인다. 또한 스킬 검색/컬렉션 분류를 통해 문서 관리 비용을 줄인다.
**영향 분석**: 개발자 생산성 측면에서 스킬 재사용률이 올라가고 설정 드리프트가 감소한다. 스타트업은 팀 단위로 동일 스킬을 표준화해 온보딩이 쉬워진다. 인디 빌더에게는 개인 PC 여러 툴 환경을 안정적으로 유지하는 데 유리하다. eastsea 시스템에서는 여러 모델/툴을 실험할 때 설정 동기화 실수를 줄이는 데 직접적 이득이 있다.
**Master 액션 포인트**: 1) 개인 장비에서 Claude Code와 관련 스킬 저장소를 정기 동기화하고, Chops로 탐색 가능한 구조로 정비한다. 2) 팀 공통 스킬을 컬렉션 단위로 분류해 버전 관리/공유를 표준화한다.
