### 1. legalize-kr — 법령을 코드처럼, 판례도 코드처럼 (69pts)
→ 원문: [Legalize — 법령과 판례를 코드처럼](https://legalize.kr)
→ 교차확인: [국가법령정보 공동활용 LAW OPEN DATA](https://open.law.go.kr/LSO/main.do)

**요약**: legalize-kr는 대한민국 법령과 판례를 PDF나 폐쇄형 뷰어가 아니라 Markdown과 Git 이력으로 다루게 만드는 프로젝트입니다. 사이트 설명대로 각 법령은 Markdown 파일로 구조화되고, 각 개정은 실제 날짜를 가진 커밋처럼 기록됩니다. 판례 역시 사건 종류와 법원 등급별로 Markdown 파일로 정리되어 grep, git log, git diff 같은 익숙한 개발 도구로 탐색할 수 있습니다. 핵심 가치는 단순 검색이 아니라 “무엇이 언제 어떻게 바뀌었는지”를 diff 중심으로 추적하게 만든 점입니다. 법령과 판례를 코드처럼 다루면 규제 해석, 제품 약관 검토, 정책 변경 감시가 훨씬 자동화 친화적으로 바뀝니다.

**기술적 배경**: 이 프로젝트가 지금 주목받는 이유는 법률 데이터 자체보다 **법률 데이터를 다루는 인터페이스**를 바꿨기 때문입니다. 기존 국가 법령 서비스는 공식성이 강하지만 개발 파이프라인과의 결합이 약했고, 변경 추적도 사람 중심 UI에 묶여 있었습니다. legalize-kr는 공식 OpenAPI를 원천으로 삼으면서도 Git 기반 버전 관리 계층을 얹어 개발자 워크플로와 바로 접속시켰습니다.

**영향 분석**: 개발자에게는 규정 변경 모니터링과 서비스 정책 검토가 스크립트 가능한 문제로 바뀝니다. 스타트업에게는 개인정보, 전자상거래, 노동, 세무 관련 규제 리스크를 제품 개발 루프 안으로 더 일찍 끌어올릴 수 있다는 의미가 있습니다. 인디 빌더에게도 앱 심사, 약관, 정책 페이지, 고객지원 응답을 근거 기반으로 만드는 토대가 됩니다.

**Master 액션 포인트**:
- OpenClaw 크론으로 `개인정보보호법`, `전자상거래법`, `게임산업진흥법` 등 핵심 법령 변경을 주간 diff 리포트로 받는 자동화 파일럿을 만들 가치가 있습니다.
- eastsea.xyz의 정책/약관 문서를 Git 관리 법령 저장소와 연결해, 변경 시 관련 페이지 점검 체크리스트를 자동 생성하도록 묶는 것이 좋습니다.

### 2. 코드를 읽기 전에 실행하는 Git 명령들 (75pts)
- 원문: [The Git Commands I Run Before Reading Any Code](https://piechowski.io/post/git-commands-before-reading-code/)

**요약**: 이 글은 새로운 코드베이스를 열자마자 파일 내용을 읽기보다 Git 이력부터 보라는 실전 조언입니다. 작성자는 변경 빈도가 높은 파일, 버그 관련 커밋이 몰리는 파일, 주요 기여자 편중도, 월별 커밋 흐름, hotfix·rollback 패턴을 먼저 보면서 코드베이스의 통증 지점을 파악합니다. 특히 churn과 bug hotspot이 겹치는 파일을 가장 위험한 영역으로 간주하는 접근이 인상적입니다. 즉 코드의 구조는 파일 안에도 있지만, 더 선명한 진실은 히스토리 안에 있다는 주장입니다. 첫 하루를 “읽기”보다 “진단”에 쓰게 만드는 관점 전환이 핵심입니다.

**기술적 배경**: 에이전트 코딩 시대일수록 이 접근이 중요해지는 이유는 모델이 코드를 잘 생성해도 **레거시 구조적 위험**은 히스토리에서 더 잘 드러나기 때문입니다. 복잡도 지표만으로는 놓치기 쉬운 팀 이탈, 소유권 편중, 반복 파손 영역을 Git 로그가 노출합니다. 대안이 없는 것은 아니지만, 별도 분석 도구 없이 기본 Git만으로 시작할 수 있다는 점이 강점입니다.

**영향 분석**: 개발자는 코드 리뷰와 리팩터링 우선순위를 더 빨리 잡을 수 있습니다. 스타트업은 인수인계나 감사(audit) 과정에서 기술부채를 설명 가능한 데이터로 바꿀 수 있습니다. 인디 빌더에게도 “어디를 AI에게 맡기면 위험한가”를 가르는 기준이 생깁니다.

**Master 액션 포인트**:
- OpenClaw용 저장소 진단 프리셋으로 churn·bug hotspot·bus factor 스냅샷을 자동 생성하는 `/repo-audit` 스킬을 만드는 것이 유효합니다.
- eastsea-blog나 게임 저장소의 핫스팟 파일을 주간 리포트로 뽑아, 구조 개선 대상과 에이전트 자동 수정 금지 구역을 분리할 수 있습니다.

### 3. Arcmark - macOS용 브라우저 사이드북마크 관리자 (1pt)
- 원문: [Geek-1001/arcmark](https://github.com/Geek-1001/arcmark)

**요약**: Arcmark는 Arc 브라우저의 사이드바형 북마크 경험을 다른 브라우저에도 붙이는 네이티브 macOS 앱입니다. Swift와 AppKit으로 만들어졌고, Chrome·Arc·Safari·Brave 같은 브라우저 창 옆에 떠다니는 사이드바처럼 붙어 동작합니다. 북마크를 워크스페이스와 폴더 계층으로 나누고, Arc의 `StorableSidebar.json`을 읽어 기존 구조를 가져올 수 있게 한 점이 현실적입니다. 또한 로컬 JSON 파일 하나에 저장하는 로컬 우선 구조라 배포와 백업도 단순합니다. 브라우저를 바꾸더라도 작업 맥락은 유지하겠다는 욕구를 꽤 정확히 겨냥한 도구입니다.

**기술적 배경**: Arc 이후 사용자는 브라우저 그 자체보다 **정보 조직 인터페이스**에 더 익숙해졌습니다. Arcmark는 브라우저 엔진을 새로 만들지 않고, 접근성 권한과 네이티브 윈도우 제어를 활용해 UI 레이어만 재구성합니다. 완전한 브라우저 대체보다 훨씬 가볍고, 특정 UX만 떼어와 재사용하는 점이 차별점입니다.

**영향 분석**: 개발자에게는 정보 탐색 레이아웃이 브라우저 종속적일 필요가 없다는 신호입니다. 스타트업에는 “기존 플랫폼 위에 얇은 워크플로 계층을 얹는 제품”의 가능성을 보여줍니다. 인디 빌더에게도 무거운 플랫폼 구축보다 얇은 인터페이스 재조합이 더 빨리 팔릴 수 있음을 시사합니다.

**Master 액션 포인트**:
- eastsea.xyz 운영에서 리서치 링크, 배포 체크리스트, 에셋 링크를 묶는 전용 사이드카형 북마크 UI를 실험해볼 만합니다.
- 브라우저 자체를 만들기보다 생산성 워크플로에 붙는 macOS 유틸리티가 더 빠른 수익화 경로인지 재평가할 가치가 있습니다.

**요약**: agent-skills는 AI 코딩 에이전트가 스펙 작성, 계획 수립, 구현, 테스트, 리뷰, 배포 단계를 일관되게 따르도록 만든 스킬 패키지입니다. 단순 프롬프트 모음이 아니라 `/spec`, `/plan`, `/build`, `/test`, `/review`, `/ship` 같은 라이프사이클 명령으로 작업 흐름을 구조화합니다. 각 스킬은 안티-합리화 표, 검증 요구사항, 단계별 체크포인트를 포함해 “왜 테스트를 건너뛰면 안 되는가”까지 강제합니다. 여러 IDE와 에이전트 하네스에 이식 가능하다는 점도 중요한 포인트입니다. 결국 이 프로젝트는 좋은 결과를 우연에 맡기지 말고, 숙련 엔지니어의 습관을 재사용 가능한 운영 규칙으로 패키징하자는 제안입니다.

**기술적 배경**: 2026년 에이전트 시장의 문제는 모델 성능보다 **실패 패턴의 반복**입니다. agent-skills는 이 문제를 더 똑똑한 모델이 아니라 더 촘촘한 절차적 가드레일로 해결하려 합니다. 대안인 긴 시스템 프롬프트보다 유지보수와 재사용성이 높고, 특정 행동을 규약화하기 쉽다는 장점이 있습니다.

**영향 분석**: 개발자에게는 팀의 코딩 규율을 에이전트 행동으로 옮기는 출발점이 됩니다. 스타트업에게는 온보딩과 코드 품질 격차를 줄이는 운영 자산이 됩니다. 인디 빌더에게도 “내가 늘 중요하게 여기는 개발 원칙”을 에이전트에 위임할 수 있다는 점에서 생산성 복리가 큽니다.

### 5. Advisor 전략: Opus를 조언자로 활용해 Sonnet의 지능을 끌어올리기 (30pts)

**요약**: Anthropic은 작은 실행자 모델이 작업을 끝까지 끌고 가다가 중요한 의사결정 지점에서만 더 큰 모델에게 자문을 구하는 `advisor strategy`를 공식화했습니다. Sonnet 또는 Haiku가 실제 툴 호출과 반복을 담당하고, Opus는 계획·수정·중단 판단만 제공하는 구조입니다. 공개 수치상 Sonnet+Opus 조합은 Sonnet 단독 대비 SWE-bench Multilingual에서 성능을 끌어올리면서도 총비용은 낮췄고, Haiku+Opus는 훨씬 싼 비용으로 지능 상승 효과를 얻었습니다. 중요한 것은 모델을 바꾸는 것이 아니라, **언제 더 높은 지능을 호출할지**를 시스템 차원에서 최적화한다는 점입니다. 이건 단순 벤치마크 개선이 아니라 에이전트 아키텍처 표준의 변화 신호입니다.

**기술적 배경**: 기존 멀티에이전트 구조는 큰 모델이 오케스트레이션하고 작은 모델이 실행하는 방식이 많았습니다. 이번 전략은 반대로 작은 모델이 주행하고, 큰 모델은 짧고 비싼 전략 자문만 맡습니다. 이는 긴 작업의 대부분이 기계적 루프라는 현실을 인정한 설계이고, 서버 측에서 단일 API 요청 안에 모델 핸드오프를 감춘 점도 차별점입니다.

**영향 분석**: 개발자에게는 비용과 품질을 동시에 잡는 새로운 기본 패턴이 생깁니다. 스타트업은 전면 Opus 의존 없이도 복잡한 작업의 성공률을 올릴 여지가 커집니다. 인디 빌더에게는 “항상 최고 모델” 대신 “중요한 순간에만 최고 모델”이라는 가격 전략이 매우 실용적입니다.

**Master 액션 포인트**:
- OpenClaw의 장기 실행 작업에 `기본 모델 + 고지능 리뷰어` 구조를 부분 도입해, 어려운 분기에서만 상위 모델 호출이 일어나게 설계할 만합니다.
- eastsea.xyz 콘텐츠/코드 생성 파이프라인에서 초안 생성은 저비용 모델, 최종 구조 판단은 상위 모델이 맡는 2단계 비용 최적화를 실험해보는 것이 좋습니다.

### 13. Awesome Design.MD - 유명 웹사이트 디자인 시스템을 내 사이트에 적용하기 (100pts)

**요약**: Awesome Design.MD는 Google Stitch가 소개한 `DESIGN.md` 개념을 실제 유명 서비스 디자인 시스템 예시 모음으로 확장한 저장소입니다. 핵심 발상은 디자인 시스템을 JSON이나 피그마 토큰보다, 에이전트가 바로 읽기 좋은 Markdown 문서로 제공하는 것입니다. 저장소에는 브랜드별 색상, 타이포, 컴포넌트 규칙, 레이아웃 원칙, 반응형 행동, 안티패턴까지 정리된 파일들이 포함돼 있습니다. 즉 AGENTS.md가 “어떻게 만들지”를 정의한다면, DESIGN.md는 “어떤 분위기와 시스템으로 보여야 할지”를 정의합니다. AI 코딩 에이전트가 프런트엔드 구현까지 깊게 관여하는 지금, 매우 시의적절한 포맷 실험입니다.

**기술적 배경**: 기존 디자인 핸드오프는 Figma와 코드 사이에서 많은 의미 손실이 발생했습니다. DESIGN.md는 이를 텍스트 규약으로 단순화해 에이전트가 바로 소비할 수 있게 합니다. 완전 자동 생성보다 “디자인 의도를 문장으로 표준화”하는 쪽이 현재 LLM과 더 잘 맞는다는 점이 차별점입니다.

**영향 분석**: 개발자는 디자인 의사결정을 코드와 같이 버전 관리할 수 있습니다. 스타트업은 디자인팀 없이도 브랜드 일관성을 더 오래 유지할 수 있습니다. 인디 빌더에게는 템플릿 복제보다 “우리만의 디자인 규약”을 빠르게 자산화하는 수단이 됩니다.

**Master 액션 포인트**:

### 15. 나는 여전히 Skills보다 MCP를 선호한다 (5pts)
- 원문: [I Still Prefer MCP Over Skills](https://david.coffee/i-still-prefer-mcp-over-skills/)

**요약**: 이 글은 최근 AI 업계가 Skills를 새로운 표준처럼 밀고 있지만, 실제 서비스 연결과 권한 위임에는 여전히 MCP가 더 우월한 아키텍처라고 주장합니다. 작성자는 순수 지식 스킬은 유용하다고 인정하면서도, CLI 설치와 시크릿 관리, 환경 종속성, 컨텍스트 팽창을 요구하는 스킬은 범용 표준이 되기 어렵다고 봅니다. 반대로 MCP는 원격 사용, 자동 업데이트, 인증 처리, 샌드박싱, 포터빌리티 면에서 훨씬 서비스 친화적이라고 설명합니다. 결론은 “서비스 연결은 MCP, 사용법과 절차 지식은 Skill”이라는 역할 분리가 더 현실적이라는 것입니다. 지금 업계 논쟁을 정리하는 데 꽤 유효한 관점입니다.

**기술적 배경**: 2026년 에이전트 생태계는 툴 표준 전쟁의 초입에 있습니다. 스킬은 문서형 지식과 워크플로 교육에 강하고, MCP는 상태 있는 연결과 권한 위임에 강합니다. 이 글의 차별점은 양자택일보다 계층 분리를 제안한다는 점입니다.

**영향 분석**: 개발자는 어떤 기능을 문서로 가르치고, 어떤 기능을 커넥터로 노출할지 더 명확히 구분하게 됩니다. 스타트업은 CLI 남발보다 표준 인터페이스와 인증 흐름을 먼저 설계해야 합니다. 인디 빌더에게도 서비스 통합 제품을 만들 때 운영 복잡도를 크게 줄일 수 있는 사고방식입니다.

**Master 액션 포인트**:
- OpenClaw는 계속 `Skill = 절차 지식`, `Tool/MCP = 실행 인터페이스` 원칙을 강화하는 쪽이 맞습니다.
- 외부 서비스 연동을 새로 만들 때는 README형 가이드보다 먼저 원격 커넥터 계층을 설계하는 편이 장기적으로 이롭습니다.
