### 1. Cherri - 애플 단축어를 코드로 작성하는 프로그래밍 언어 (4pts)
- 원문: [electrikmilk/cherri](https://github.com/electrikmilk/cherri)
- 교차확인: [Cherri 언어 문서](https://cherrilang.org/language/)
**요약**: Cherri는 Apple Shortcuts를 그래픽 블록 편집기가 아니라 텍스트 기반 언어로 작성하고, 이를 실행 가능한 Shortcut 파일로 직접 컴파일하는 도메인 특화 언어입니다. 저장소 기준으로 1대1 액션 매핑, 타입 시스템, 함수, include, 패키지 매니저, VS Code 확장, playground, macOS IDE까지 갖춘 비교적 성숙한 생태계를 지향합니다. 특히 “디버깅이 쉬워야 한다”는 의식이 강해서 가능한 한 Shortcuts 액션과 소스 코드의 대응 관계를 보존하려는 점이 눈에 띕니다. 이 프로젝트가 흥미로운 이유는 단축어가 오래전부터 강력했지만 GUI 중심 저작 방식 때문에 대규모 유지보수 자산으로 다루기 어려웠다는 구조적 문제를 정면으로 푼다는 데 있습니다. 즉 Cherri는 단축어를 장난감 자동화가 아니라 버전 관리 가능한 코드 자산으로 끌어올리려는 시도입니다.
**기술적 배경**: Apple Shortcuts는 강하지만 규모가 커질수록 재사용, diff, 협업, 리팩터링이 매우 불편합니다. Cherri는 이 불편을 언어 설계와 컴파일러 계층으로 해결해 Shortcut을 코드 중심 워크플로우에 편입시킵니다. 기존 GUI 자동화 대비 유지보수성과 재배포성이 훨씬 좋아지는 방향입니다.
**영향 분석**: 개발자는 개인 자동화를 깔끔하게 버전 관리할 수 있고, 스타트업은 내부 반복 작업을 더 안정적으로 제품화할 수 있습니다. 인디 빌더에게도 iOS/macOS 친화 자동화를 배포 가능한 자산으로 다루는 길이 열립니다.
**Master 액션 포인트**: OpenClaw 쪽 iPhone 단축어 실험이 필요해지면 Cherri 기반으로 예제 자산을 코드화해두는 편이 장기적으로 낫습니다. eastsea 콘텐츠 자동화나 카메라 앱 보조 흐름을 Shortcut으로 붙일 때도 GUI 수작업보다 이 계층이 더 유지보수 친화적입니다.

### 2. 프로그래밍의 일곱 가지 원형 언어 (2022) (39pts)
- 원문: [The seven programming ur-languages](https://madhadron.com/programming/seven_ur_languages.html)
- 교차확인: [Programming paradigm 개요](https://en.wikipedia.org/wiki/Programming_paradigm)
**요약**: 이 글은 자바, 파이썬, C++ 같은 언어 이름보다 더 아래층에 있는 사고 패턴을 보자고 제안하며, 현대 프로그래밍을 ALGOL, Lisp, ML, Self, Forth, APL, Prolog의 일곱 원형으로 분해합니다. 핵심 주장은 문법 차이는 피상적이고, 반복, 재귀, 메시지 전달, 매크로, 벡터화, 논리 추론 같은 패턴 묶음이 진짜 학습 비용을 결정한다는 점입니다. 그래서 같은 원형 안에서의 언어 이동은 비교적 쉽지만, 익숙하지 않은 원형으로 건너갈 때는 새로운 신경 회로를 만들어야 하므로 학습 난도가 급격히 올라간다고 설명합니다. 이 글이 계속 회자되는 이유는 AI 시대에도 언어 선택 논쟁보다 사고 패턴의 폭이 더 중요하다는 감각을 잘 짚기 때문입니다. 특히 에이전트가 코드 초안을 쉽게 써주는 시대일수록, 인간은 원형 수준의 추상 사고를 더 강하게 쥐고 있어야 방향을 잃지 않습니다.
**기술적 배경**: 많은 실무 언어는 결국 ALGOL 계열 표면 문법을 공유하기 때문에 서로 다르게 보여도 사고 방식은 유사합니다. 반면 함수형, 논리형, 배열 지향, 메시지 지향 계열은 문제 분해 방식 자체가 달라집니다. 그래서 좋은 개발자는 언어 수집가보다 원형 전환이 가능한 사람에 가깝습니다.
**영향 분석**: 개발자는 새 언어를 배울 때 도구보다 사고 패턴을 먼저 익히는 편이 효율적입니다. 스타트업은 팀 기술 선택에서 유행보다 문제 유형에 맞는 사고 모델을 택해야 합니다. 인디 빌더에게도 언어 갈아타기보다 적절한 추상화 습관을 늘리는 쪽이 더 큰 레버리지입니다.
**Master 액션 포인트**: OpenClaw와 게임 파이프라인 설계에서 “언어 선택” 논의가 나오면 먼저 상태기계형, 데이터흐름형, 규칙형 중 어느 문제인지부터 분해하는 편이 좋습니다. 스택 논쟁 전에 사고 모델을 고정하면 시행착오가 줄어듭니다.

### 15. DigitalOcean에서 Hetzner로 마이그레이션하기 (14pts)

**영향 분석**: 개발자는 PaaS 편의성이 정말 필요한지 다시 따져보게 됩니다. 스타트업은 고정비 절감이 곧 runway 연장으로 이어질 수 있습니다. 인디 빌더에게도 일정한 워크로드는 전용 서버가 다시 유효한 선택지가 될 수 있습니다.
**Master 액션 포인트**: eastsea 확장 시 무조건 관리형 서비스로 가지 말고 steady-state 구간은 저비용 전용 서버 조합도 비교해야 합니다. 다만 자동 복구와 운영 관제 없이 성급히 옮기면 절감보다 장애 비용이 커질 수 있습니다.

- **위험 신호 2**: AI가 코드를 더 많이 써줄수록 인간의 구조 판단력과 리뷰 품질을 유지하는 장치가 없으면 기술 부채가 더 빨리 축적됩니다.
