### 1. 기술 부채, 인지 부채, 의도 부채 (24pts)
**[기술 부채, 인지 부채, 의도 부채](https://martinfowler.com/fragments/2026-04-02.html)**
→ 원문: [Fragments: April 2](https://martinfowler.com/fragments/2026-04-02.html)
→ 교차확인: [From Technical Debt to Cognitive and Intent Debt](https://arxiv.org/abs/2603.22106)
**요약**: Martin Fowler는 생성형 AI가 코드를 대량 생산하는 시대에는 전통적인 기술 부채만으로 시스템 건강을 설명하기 어렵다고 짚었습니다. 여기서 핵심은 코드에 쌓이는 기술 부채 외에도, 팀의 공유 이해가 약해지는 인지 부채와 목표·제약·이유가 문서화되지 않는 의도 부채가 함께 쌓인다는 점입니다. 인지 부채는 사람이 시스템을 안전하게 바꿀 수 있는 능력을 갉아먹고, 의도 부채는 인간과 에이전트가 모두 왜 이런 구조가 되었는지 이해하지 못하게 만듭니다. 원문이 흥미로운 이유는 이 세 가지 부채가 서로 따로가 아니라 연쇄적으로 악화된다고 본다는 점입니다. 코드가 많아질수록 이해가 약해지고, 이해가 약해질수록 의도가 사라지며, 의도가 사라질수록 다음 변경이 더 나빠지는 구조입니다. 결국 AI 시대의 리스크는 더 많은 코드가 아니라, 더 적은 이해와 더 적은 설명일 수 있습니다.
**기술적 배경**: 이 논의는 최근 소프트웨어 건강을 코드 품질만으로 보지 말자는 흐름과 맞닿아 있습니다. 특히 에이전트 코딩이 빨라질수록 ADR, 작업 이유, 제약 조건, 테스트 의도 같은 외부화된 지식이 사실상 런타임 의존성처럼 중요해집니다. 기존 대안이 기술 부채 감축을 리팩터링 중심으로 다뤘다면, 이 프레임은 사람과 문서 계층까지 운영 대상으로 끌어올린다는 점이 차별점입니다.
**영향 분석**: 개발자는 이제 코드 리뷰만 잘해도 충분하지 않습니다. 스타트업은 작은 팀일수록 암묵지가 사라질 때 타격이 더 크기 때문에, AI 도입 속도가 빠를수록 설계 이유와 완료 기준을 더 자주 기록해야 합니다. 인디 빌더에게도 문서화는 귀찮은 부가 작업이 아니라, 미래의 자신과 미래의 에이전트를 위한 실행 자산이 됩니다.
**Master 액션 포인트**: OpenClaw 구현 세션마다 `Current state / Verification criteria / Completion criteria / Artifact path`를 남기는 현재 규율을 더 강하게 자동화해야 합니다. eastsea와 게임 파이프라인에서도 큰 기능마다 5줄짜리 의도 메모와 실패 조건을 남겨, 의도 부채가 구조적으로 쌓이지 않게 막는 편이 맞습니다.

### 2. 나는 클라우드를 만들고 있어요 (12pts)
**[나는 클라우드를 만들고 있어요](https://crawshaw.io/blog/building-a-cloud)**
**요약**: Tailscale 공동창업자 David Crawshaw는 오늘날 클라우드의 기본 추상화가 개발자에게 너무 불리한 모양이라고 정면으로 비판합니다. CPU, 메모리, 디스크를 원하는 대로 조합해 쓰는 대신, VM과 PaaS가 공급자 정의 한계 안에서만 움직이게 만들어 실제 컴퓨터보다 덜 자유롭다는 주장입니다. 특히 원격 블록 스토리지와 높은 egress 비용, 계층이 지나치게 많은 API가 현대 클라우드의 핵심 비효율로 지목됩니다. 그가 새로 만드는 exe.dev는 개별 VM이 아니라 CPU와 메모리 풀을 주고 그 위에서 여러 머신을 운영하게 하며, 로컬 NVMe와 anycast 네트워크를 기본으로 둡니다. 같은 날 공개된 Series A 발표를 보면 이 회사는 이를 위해 직접 데이터센터 장비를 사서 클라우드 1.0의 기본값을 다시 설계하겠다고 밝힙니다. AI 에이전트가 늘어날수록 더 많은 소프트웨어가 생기는데, 그 소프트웨어가 살 집이 지금 클라우드는 아니라는 문제의식이 분명합니다.
**기술적 배경**: 기존 대형 클라우드는 범용성과 매출 극대화에 최적화되어 있고, 그 결과 디스크 IOPS, 네트워크 egress, VM 모양새 자체가 공급자 편의에 맞게 굳었습니다. exe.dev는 이를 SSH, 프라이빗 기본값, 단순 과금, 직접 장비 운영으로 뒤집으려 합니다. Heroku가 가졌던 단순성과 VM 수준 제어를 동시에 가져오려는 시도라는 점이 지금 주목받는 이유입니다.
**영향 분석**: 개발자는 이제 단순히 AWS를 배우는 것이 아니라, 어떤 추상화가 정말 필요한지 다시 생각하게 됩니다. 스타트업은 AI와 자동화 때문에 더 많은 내부 툴과 마이크로서비스를 만들게 되므로, 인프라 기본값의 비용 왜곡이 더 빨리 드러날 수 있습니다. 인디 빌더에게도 저렴하고 단순한 컴퓨트는 실험 횟수를 직접 늘려 주는 레버입니다.
**Master 액션 포인트**: OpenClaw와 게임 파이프라인 배포는 가능한 한 SSH, 표준 Linux, 간단한 파일/프로세스 모델 위주로 유지하는 편이 좋습니다. eastsea 인프라 실험도 “더 많은 매니지드 서비스”보다 “더 적은 추상화로 더 예측 가능한 운영”을 우선순위로 두는 쪽이 맞습니다.

### 3. LLM으로 할 수 있는 비전형적인 일 7가지 (6pts)
**[LLM으로 할 수 있는 비전형적인 일 7가지](https://www.kdnuggets.com/7-specific-unconventional-things-to-do-with-language-models)**
**요약**: 이 글은 LLM을 단순 채팅창으로 소비하는 대신, 의사결정 반박자, 난해한 에러 해설자, 계약 검토 보조, 역할극 기반 브레인스토머, 러버덕 디버깅 파트너, 개인 맞춤 학습 코치, 문화적 맥락 번역기로 쓰는 방법을 정리합니다. 핵심은 각각의 사용법이 모델 능력보다 프롬프트 구조를 바꿔 사고의 틀을 전환한다는 데 있습니다. 예를 들어 “동의하는 조수” 대신 “논리적 반박자”로 역할을 지정하면, 검증 편향을 줄이는 보조 도구로 변합니다. 또 에러 로그 해설과 러버덕 디버깅은 모델을 검색엔진이 아니라 사고 정리 장치로 쓰는 예입니다. 법률 문서 요약이나 문화적 뉘앙스 해설은 프라이버시와 맥락 민감성이 높은 도메인에서 특히 의미가 큽니다. 결국 이 글은 LLM의 가치는 질문 답변보다 작업 틀 전환에 있다는 점을 대중적으로 잘 보여 줍니다.
**기술적 배경**: 최근 에이전트 담론은 거대한 자율 실행에 치우치기 쉬운데, 실제 현장에서는 짧고 선명한 역할 기반 프롬프트가 더 높은 실용성을 보일 때가 많습니다. 이 글은 별도 툴 체인 없이도 인지 보조 계층을 늘릴 수 있다는 점에서 가볍지만 강합니다. 기존 대안이 “요약해 줘” 수준에 머무른다면, 여기서는 사고 모드 자체를 바꾸는 템플릿을 제안합니다.
**영향 분석**: 개발자는 LLM을 코드 생성기 하나로만 쓰지 않고, 설계 반박자와 사고 디버거로 분리해서 사용할 수 있습니다. 스타트업은 고객지원, 영업, 계약 검토, 사내 문서 해석 같은 비개발 워크플로까지 LLM 활용 폭을 넓힐 수 있습니다. 인디 빌더에게는 작은 제품에서도 차별화된 UX를 만드는 프롬프트 패턴 라이브러리의 가치가 커집니다.
**Master 액션 포인트**: OpenClaw에 `반박`, `러버덕`, `계약/약관 요약`, `문화 맥락 설명` 같은 경량 스킬을 추가하면 활용 밀도가 올라갑니다. eastsea 콘텐츠도 “AI로 코드를 쓴다”보다 “AI로 사고를 구조화한다” 쪽이 더 오래 가는 주제로 보입니다.

### 4. syntaqlite - SQLite 자체 문법과 토크나이저 기반의 SQL 파서·포매터·검증기·언어 서버 (10pts)
**[syntaqlite - SQLite 자체 문법과 토크나이저 기반의 SQL 파서·포매터·검증기·언어 서버](https://github.com/lalitMaganti/syntaqlite)**
**요약**: syntaqlite는 SQLite를 대충 흉내 내는 범용 SQL 파서가 아니라, SQLite의 Lemon 문법과 토크나이저를 직접 재사용해 정확도를 끌어올린 도구입니다. 프로젝트 설명에 따르면 약 39만 6천 개의 upstream 테스트 구문에서 99.7% 수준의 파싱 일치를 보였고, 비교 문서에서는 SQLite 특화 40개 구문 테스트에서 100% 정확도를 내세웁니다. 중요한 점은 단순 파싱만 하는 것이 아니라 포매터, 정적 검증기, LSP, Python/JS 문자열 내 SQL 분석까지 한 묶음으로 제공한다는 것입니다. 또 SQLite 버전과 컴파일 플래그를 지정해 실제 배포 환경과 같은 문법 표면을 맞출 수 있습니다. Android처럼 최신 SQLite가 아닌 환경에서도 `RETURNING` 같은 최신 문법 오판을 줄여 준다는 것이 실전적입니다. SQLite를 많이 쓰는 모바일, 임베디드, 로컬 우선 제품에서 꽤 강한 개발자 경험 향상을 줄 수 있는 도구입니다.
**기술적 배경**: 대부분의 SQL 도구는 범용 파서를 만든 뒤 SQLite를 하나의 flavor로 취급합니다. 하지만 SQLite는 fallback token, compile-time flag, 버전별 기능 차이까지 포함하면 훨씬 까다로운 언어이므로 이 접근이 자주 깨집니다. syntaqlite는 SQLite가 실제로 사용하는 생성기인 Lemon과 원문법을 그대로 가져와 이 간극을 줄인다는 점이 차별 포인트입니다.
**영향 분석**: 개발자는 런타임에서만 터지던 SQL 문제를 더 이른 단계에서 잡을 수 있습니다. 스타트업은 로컬 DB 기반 제품의 정적 검증 품질을 올리면서도 무거운 서버형 데이터베이스에 의존하지 않을 수 있습니다. 인디 빌더에게는 SQLite가 “작지만 위험한 DB”가 아니라 “정적 도구를 갖춘 안정적 기본값”으로 올라갈 여지가 생깁니다.

**기술적 배경**: 최근 코딩 에이전트는 기능이 늘면서 한 세션 안에 제품 판단, 구현, 리뷰, QA, 배포를 모두 억지로 얹는 경우가 많습니다. gstack류 접근은 이를 명시적 역할과 강제 순서로 분해합니다. 기존 대안이 범용 프롬프트 장인정신에 머물렀다면, Skillify는 구조적 재발 방지라는 점에서 한 단계 더 운영 지향적입니다.
**영향 분석**: 개발자는 이제 좋은 AI 사용법이 곧 좋은 작업 분해 능력이라는 사실을 더 분명히 느끼게 됩니다. 스타트업은 적은 인원으로 더 많이 만들 수 있지만, 그 전제는 역할을 혼합하지 않는 강한 프로토콜입니다. 인디 빌더에게도 slash-command식 운영 규율은 작은 팀에서 특히 효과가 큽니다.
**Master 액션 포인트**: OpenClaw는 이미 스킬과 서브에이전트 구조가 있으니, 이를 더 명시적인 `/plan`, `/review`, `/qa`, `/release` 경로로 표준화하는 편이 좋습니다. eastsea에서는 에이전트 시대의 생산성 본질을 “모델 성능”보다 “역할 구조 설계”로 정리한 심층 글을 별도로 파도 좋습니다.

### 15. vLLM Recipes 개편 - 모델+하드웨어 조합별 설정을 딸각 한방으로 (16pts)
**[vLLM Recipes 개편 - 모델+하드웨어 조합별 설정을 딸각 한방으로](https://recipes.vllm.ai/)**
**요약**: 새 `vLLM Recipes`는 “어떤 모델을 어떤 GPU 조합에서 어떻게 띄우는가”를 인터랙티브하게 고르게 하고, 곧바로 실행 명령을 복사하게 해 주는 설정 가이드입니다. 사이트 설명만 봐도 H100, H200, B200, B300, Grace Blackwell, AMD MI300 계열처럼 실제 배포 현장에서 많이 부딪히는 하드웨어 축을 전면에 둡니다. 이는 LLM 서빙의 병목이 모델 선택 자체보다, 모델과 하드웨어와 serve flag 조합을 아는 지식 비용에 있다는 현실을 반영합니다. vLLM 메인 사이트가 강조하듯 이 프로젝트의 강점은 PagedAttention, 연속 배칭, OpenAI 호환 API, 다양한 하드웨어 지원에 있습니다. Recipes는 그 기술적 강점을 “복붙 가능한 운영 지식”으로 포장하는 레이어입니다. 즉 서빙 엔진 경쟁이 이제 성능표뿐 아니라 배포 지식의 제품화로 넘어가고 있다는 신호입니다.
**기술적 배경**: 자가 호스팅 LLM은 종종 모델보다 실행 파라미터와 GPU 메모리 전략에서 더 많이 실패합니다. vLLM Recipes는 그 시행착오를 템플릿화해 진입 장벽을 낮춥니다. 기존 대안이 scattered docs나 Discord 답변에 흩어져 있었다면, 이 도구는 조합형 운영 지식을 공식 인터페이스로 끌어올립니다.
**영향 분석**: 개발자는 이제 오픈 모델 서빙이 “전문가만 하는 일”에서 점점 복제 가능한 운영 메뉴얼로 변하고 있음을 느낄 수 있습니다. 스타트업은 GPU 운용 인력이 적어도 특정 조합의 배포 가능성을 더 빨리 검토할 수 있습니다. 인디 빌더에게는 아직 고급 하드웨어 진입장벽이 높지만, 운영 지식이 표준화될수록 시장 진입 비용은 점점 내려갑니다.
**Master 액션 포인트**: MiniPC, NAS, 원격 GPU 후보를 평가할 때 하드웨어별 serve 레시피를 축적하는 내부 노트가 필요합니다. eastsea에서는 “오픈 모델을 실제로 띄우는 지식” 자체를 콘텐츠 자산과 운영 자산으로 함께 쌓는 편이 좋습니다.
