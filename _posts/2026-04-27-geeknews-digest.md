### 1. Obscura - 오픈소스 헤드리스 브라우저 (2pts)
**요약**: Obscura는 Rust로 작성된 초경량 헤드리스 브라우저 엔진으로, 스크래핑과 AI 에이전트 자동화에 특화돼 있습니다. 원문 README는 V8을 내장해 실제 JavaScript를 실행하고 Chrome DevTools Protocol을 구현해 Puppeteer와 Playwright의 드롭인 대체재처럼 쓸 수 있다고 설명합니다. 프로젝트가 강조하는 차별점은 메모리 사용량 약 30MB, 빠른 기동, 단일 바이너리, 기본 내장 스텔스 모드입니다. 즉 기존 headless Chrome 계열이 무거운 런타임과 탐지 표면을 안고 있었다면, Obscura는 그 비용을 엔진 단계에서 줄이려는 시도입니다. 아직 초기 프로젝트이지만, “브라우저 자동화 = Chrome 전체를 띄워야 한다”는 가정을 흔드는 흥미로운 출발점입니다.
**기술적 배경**: 최근 에이전트 워크플로는 웹 조작 비중이 커졌지만, 실행 비용과 탐지 회피는 여전히 큰 병목입니다. Obscura는 기존 브라우저 위에 자동화 레이어를 얹는 대신, 자동화 친화적 브라우저 자체를 새로 만드는 방향이라는 점이 차별점입니다.
**영향 분석**: 개발자는 단순 수집기나 에이전트용 브라우저 런타임을 더 작게 배포할 수 있다는 가능성을 보게 됩니다. 스타트업과 인디 빌더에게는 무거운 Chrome 기반 수집 스택 대신 더 싼 런타임 옵션이 생길 수 있습니다.
**Master 액션 포인트**: OpenClaw의 웹 작업 중 대량 스크래핑 계열은 장기적으로 Obscura류 경량 런타임을 벤치마크할 가치가 있습니다. eastsea에는 “AI 에이전트 시대의 브라우저는 Chrome 래퍼보다 전용 엔진이 유리한가”라는 비교 글감이 생겼습니다.

### 2. DESIGN.md — AI 코딩 도구를 위한 디자인 시스템 단일 파일 포맷 (한국어 정리) (44pts)
→ 원문: [DESIGN.md 포맷 완전정복](https://rubric.im/curriculum/design-md)
→ 교차확인: [google-labs-code/design.md](https://github.com/google-labs-code/design.md)
**요약**: 이 글은 Google Labs가 제안한 DESIGN.md 포맷을 한국어 커리큘럼으로 정리한 장문 해설입니다. 핵심은 디자인 토큰을 YAML 프런트매터로, 의도와 사용 원칙을 마크다운 본문으로 함께 담아 에이전트가 디자인 시스템을 지속적으로 이해하게 만드는 데 있습니다. 공식 저장소도 이 포맷을 “visual identity를 coding agents에게 설명하는 규격”으로 소개하며, lint, diff, export, spec 같은 CLI까지 제공합니다. 즉 디자인 시스템이 더 이상 Figma 참고자료나 인간용 위키가 아니라, 코드 생성 에이전트가 직접 소비하는 실행 입력으로 바뀌는 흐름입니다. AI가 UI를 만들고 수정하는 빈도가 높아질수록 이 형식의 가치는 더 커질 가능성이 높습니다.
**기술적 배경**: 기존 디자인 시스템은 토큰 JSON, 문서, 디자이너 설명이 분리된 경우가 많아 에이전트가 의도까지 안정적으로 읽기 어려웠습니다. DESIGN.md는 토큰과 설명을 한 파일로 묶고 lint 규칙과 export 경로까지 제공해, 사람과 모델 모두를 위한 단일 진실 공급원(single source of truth)을 노립니다.
**영향 분석**: 개발자는 앞으로 UI 자동화 품질을 높이려면 컴포넌트 코드보다 먼저 문서 형식을 정비해야 합니다. 스타트업은 디자이너-개발자-에이전트 사이의 해석 손실을 줄일 수 있고, 인디 빌더는 작은 팀에서도 일관된 UI 스타일을 유지하기 쉬워집니다.
**Master 액션 포인트**: OpenClaw와 eastsea의 웹 표면 공통 규칙을 `DESIGN.md` 유사 포맷으로 정리하면 자동 수정 정확도를 높일 수 있습니다. 게임 랜딩 페이지도 색상, 타이포, 버튼 규칙을 이런 형식으로 고정해 두면 UI 생성 실험 비용이 줄어듭니다.

### 3. Agentic Inbox - AI 에이전트 탑재 셀프호스팅 이메일 클라이언트 (1pt)
**요약**: Agentic Inbox는 Cloudflare Workers 위에서 돌아가는 셀프호스팅 이메일 클라이언트로, 메일함 관리와 AI 에이전트 보조를 하나의 앱으로 묶습니다. GitHub 원문은 각 메일함을 Durable Object와 SQLite로 격리하고, 첨부 파일은 R2에 저장하며, Workers AI와 Agents SDK로 초안 작성과 검색, 응답 보조를 제공한다고 설명합니다. Cloudflare 블로그는 여기에 더해 이메일이 에이전트의 핵심 인터페이스가 되고 있으며, 비동기적으로 일을 처리하고 나중에 답하는 에이전트가 실제 서비스 형태에 더 가깝다고 강조합니다. 요컨대 이 프로젝트는 “챗봇형 에이전트”를 메일 클라이언트라는 오래된 인터페이스에 현실적으로 이식한 참조 구현입니다. 채널을 새로 만들기보다 이미 모두가 쓰는 이메일을 에이전트 작업면으로 삼는 접근이 꽤 실용적입니다.
**기술적 배경**: 많은 에이전트 데모는 대화창 안에서만 머물지만, 실제 운영은 이메일, 인증, 저장소, 권한 관리가 더 어렵습니다. Agentic Inbox는 Cloudflare의 Email Routing, Access, Workers AI, Durable Objects를 묶어 그 운영층까지 보여 준다는 점이 차별점입니다.
**영향 분석**: 개발자는 에이전트 기능을 새 앱으로 만들지 않고 기존 업무 채널에 심는 방식을 더 진지하게 보게 됩니다. 스타트업은 고객 지원, 인보이스, 비동기 협업 흐름을 이메일 기반 에이전트로 전환할 여지가 커집니다.
**Master 액션 포인트**: OpenClaw 자동화도 장기적으로는 메신저뿐 아니라 이메일 워크플로를 수용하는 쪽이 유리합니다. eastsea에는 “에이전트의 UI는 채팅이 아니라 이메일일 수 있다”는 분석 글을 만들 만합니다.
- 원문: [Agentic Inbox GitHub](https://github.com/cloudflare/agentic-inbox)

### 5. mini-swe-agent - 100줄짜리 AI 에이전트로 GitHub 이슈 해결 및 커맨드라인 지원 (9pts)
**요약**: mini-swe-agent는 SWE-bench와 SWE-agent를 만든 팀이 내놓은 초경량 코딩 에이전트입니다. GitHub와 공식 문서는 이 프로젝트가 100줄 안팎의 agent core, bash-only 실행, 선형 히스토리, 독립적인 `subprocess.run` 호출을 통해 높은 단순성과 재현성을 추구한다고 설명합니다. 그럼에도 SWE-bench verified에서 74% 이상 성능을 냈고, 다양한 모델과 샌드박스 환경에서 쉽게 돌릴 수 있다고 주장합니다. 핵심 메시지는 “더 많은 툴과 더 복잡한 scaffold가 좋은 에이전트를 보장하지 않는다”는 점입니다. 에이전트가 점점 제품화되는 시점에 기본 구조를 과감히 덜어낸 사례라서 상징성이 큽니다.
**기술적 배경**: 2024년형 코딩 에이전트는 상태 유지 셸, 복잡한 history processor, 커스텀 tool interface에 크게 의존했습니다. mini-swe-agent는 모델이 강해진 지금, 에이전트 프레임워크는 오히려 얇아질 수 있다는 반론을 던집니다.
**영향 분석**: 개발자는 자기 에이전트 스택의 복잡성이 진짜 성능에 기여하는지 다시 점검하게 됩니다. 스타트업과 인디 빌더에게는 유지비가 낮고 디버깅 가능한 기본 에이전트 구조가 더 매력적인 선택지가 됩니다.
**Master 액션 포인트**: OpenClaw 일부 자동화는 복잡한 상태 기계 대신 `짧은 프롬프트 + 결정론적 스크립트 + 검증`으로 축소 실험해 볼 만합니다. eastsea에는 “에이전트는 복잡성보다 검증 가능성이 성능을 만든다”는 관점의 분석 글이 바로 나옵니다.

### 6. GoScrapy - Go기반 초고속 웹 스크래핑 프레임워크 (23pts)
**요약**: GoScrapy는 Python Scrapy의 구조를 Go 생태계로 옮겨 온 고성능 스크래핑 프레임워크입니다. README 기준으로 스파이더, 스케줄러, 워커, 파이프라인, middleware, telemetry까지 포함해 단순 라이브러리가 아니라 운영형 수집 시스템에 가깝습니다. CSS/XPath 선택자, 재시도, 쿠키 세션 분리, CLI scaffolding, CSV/JSON/MongoDB/Google Sheets/Firebase 출력 등도 기본 제공됩니다. 중요한 점은 단순히 “Go라서 빠르다”가 아니라, Scrapy 수준의 작업 모델을 Go 동시성과 함께 제공한다는 데 있습니다. Python 일변도였던 크롤링 프레임워크 층에 구조적 대안이 생겼다는 의미가 있습니다.
**기술적 배경**: Go 생태계에는 Colly 같은 얇은 라이브러리는 있었지만 Scrapy급 상위 프레임워크는 상대적으로 약했습니다. GoScrapy는 수집 파이프라인의 표준 구조를 언어 레벨 성능과 함께 가져가려는 시도라는 점이 차별점입니다.
**영향 분석**: 개발자는 대규모 수집 작업에서 Python만을 기본값으로 보지 않아도 됩니다. 스타트업은 배포 단순성과 운영 비용을 이유로 Go 기반 스택을 더 적극적으로 검토할 수 있습니다.
**Master 액션 포인트**: eastsea 수집 파이프라인이 커지면 Python 스크립트군과 GoScrapy를 같은 작업셋으로 벤치마크해 보는 편이 좋습니다. OpenClaw도 반복 수집 작업은 재시도와 관측성이 내장된 구조로 승격할 여지가 있습니다.

### 13. Show GN: Clarc - 비개발자 동료들도 Claude Code 쓸 수 있게 만든 macOS 앱 (2pts)
**요약**: Clarc는 Claude Code CLI를 그대로 쓰되, 터미널 장벽을 GUI로 덮어 비개발자나 라이트 유저도 접근할 수 있게 만든 네이티브 macOS 앱입니다. ### 14. Tolaria - Markdown 지식 베이스를 관리하는 오픈소스 macOS 앱 (16pts)
**요약**: Tolaria는 Markdown 기반 지식베이스를 관리하는 Mac/Linux 데스크톱 앱으로, second brain과 팀 문서, AI 컨텍스트 저장소를 주요 사용처로 내세웁니다. GitHub 원문은 files-first, git-first, offline-first, zero lock-in을 핵심 철학으로 강조하며, 데이터가 앱이 아니라 일반 Markdown 파일과 Git 저장소에 남는 구조를 전면에 둡니다. 특히 “AI-first but not AI-only”라는 문구가 인상적입니다. AI와 잘 붙는 구조를 만들되 특정 공급자나 제품 포맷에 종속되지는 않겠다는 태도이기 때문입니다. 지식관리 도구가 다시 예쁜 UI보다 데이터 주권과 장기 보존성으로 회귀하는 흐름을 잘 보여 줍니다.
**기술적 배경**: 최근 노트 앱은 클라우드 동기화와 폐쇄형 포맷에 의존하는 경우가 많았습니다. Tolaria는 Tauri/React 기반 앱을 쓰면서도 저장 형식은 철저히 표준 Markdown과 Git에 기대어, 에이전트와 사람 모두가 읽기 쉬운 지식 자산을 지향합니다.
**영향 분석**: 개발자는 지식 저장소를 더 쉽게 에이전트 컨텍스트로 재활용할 수 있습니다. 스타트업과 인디 빌더에게는 장기적으로 가장 비싼 자산이 데이터 형식과 구조라는 점을 다시 보여 줍니다.
**Master 액션 포인트**: OpenClaw 메모리와 운영 절차가 파일 기반이라는 점은 매우 올바른 선택입니다. eastsea에는 “AI 시대 지식베이스의 핵심은 예쁜 앱이 아니라 포맷 주권”이라는 메시지가 잘 맞습니다.

### 15. OpenAI, API에 GPT-5.5와 GPT-5.5 Pro 출시 (5pts)
**요약**: OpenAI는 4월 24일 changelog를 통해 GPT-5.5와 GPT-5.5 Pro를 API에 공개했습니다. changelog와 모델 페이지 기준으로 GPT-5.5는 105만 토큰 컨텍스트, 12.8만 출력 토큰, medium 기본 reasoning, 컴퓨터 사용, MCP, Skills, hosted shell, apply patch, web search 등 매우 넓은 툴 표면을 지원합니다. 단순히 성능이 조금 좋아진 모델이 아니라, 장기 워크플로와 도구 사용을 전제로 한 실전형 프런티어 모델로 포지셔닝된 셈입니다. 다만 가격은 GPT-5.4 대비 더 높고, 장문 입력 구간에서는 추가 과금 규칙도 붙습니다. 즉 성능과 컨텍스트는 커졌지만, 운영 설계와 비용 감각이 더 중요해졌다는 의미이기도 합니다.
**기술적 배경**: 최근 모델 경쟁은 IQ 자체보다 긴 컨텍스트, 안정적인 툴 호출, 장기 세션 유지, 캐시 정책으로 옮겨가고 있습니다. GPT-5.5는 그 흐름을 그대로 반영한 모델이며, 특히 응답 API 기반 에이전트 구축에 필요한 기능이 촘촘히 붙어 있다는 점이 차별점입니다.
**영향 분석**: 개발자는 더 긴 세션과 복합 툴 워크플로를 설계하기 쉬워집니다. 반면 스타트업과 인디 빌더는 비용, 캐시 전략, 어떤 작업에 굳이 5.5를 쓸지에 대한 라우팅 정책이 더 중요해집니다.
**Master 액션 포인트**: OpenClaw는 작업 유형별 모델 라우팅 기준을 더 세밀하게 남겨 비용 대비 효율을 최적화할 필요가 있습니다. eastsea에는 “GPT-5.5는 더 똑똑한 모델이 아니라 더 긴 워크플로용 운영 모델”이라는 관점의 해설이 유효합니다.
