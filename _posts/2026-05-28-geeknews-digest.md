## 주요 이슈

### 1. Decepticon - 레드팀을 위한 자율 해킹 에이전트 (4pts)
**요약**: Decepticon은 전통적인 취약점 스캐너가 아니라, 실제 공격자처럼 정찰·익스플로잇·권한 상승·횡적 이동까지 이어지는 자율형 레드팀 체인을 지향하는 프로젝트입니다. 공개 문서는 이 도구를 단순 LLM 래퍼가 아니라 Red Team Testing 전용 자율 해킹 에이전트로 규정합니다. 핵심 메시지는 “AI 공격자는 이미 현실”이며, 이를 방어하려면 조직도 AI 기반 공격 시뮬레이터를 가져야 한다는 점입니다. 즉 제품의 포인트는 취약점 발견 자체보다, 블루팀 탐지·대응 체계를 실제 적대자 시나리오로 압박하는 데 있습니다. 보안 자동화가 이제 리포트 생성 단계를 넘어 공격 워크플로 자동화 단계로 이동하고 있음을 보여 줍니다.
**기술적 배경**: 최근 보안 AI는 정적 점검보다 문맥 이해, 공격 체인 연결, 장기 목표 유지가 더 중요해졌습니다. Decepticon은 이 흐름을 반영해 레드팀 방법론과 자율 에이전트 구조를 결합합니다.
**영향 분석**: 보안팀에는 AI 에이전트 제품과 내부 자동화를 공격자 관점에서 테스트할 수 있는 새로운 하니스 후보가 됩니다. 일반 개발팀에도 “AI 기능을 붙였으면 이제 AI 공격면도 함께 생긴다”는 신호입니다.
**Master 액션 포인트**:
- OpenClaw 계열 자동화에는 prompt injection, 권한 오남용, 데이터 유출 경로를 따로 점검하는 레드팀 체크리스트를 분리하십시오.
- 에이전트형 기능은 QA와 별개로 “장기 목표를 가진 적대자” 시나리오 검증을 붙이는 편이 안전합니다.
- 원문: [Decepticon GitHub](https://github.com/PurpleAILAB/Decepticon)
- 교차확인: [Decepticon Overview](https://docs.decepticon.red/en/introduction/overview.md)

### 2. OpenHuman - 개인용 AI 슈퍼 인텔리전스 (24pts)
**요약**: OpenHuman은 단발성 챗봇이 아니라 사용자의 이메일·문서·채팅·작업 기록을 장기적으로 흡수하는 개인용 에이전트 운영체제를 표방합니다. 핵심 차별점은 Memory Tree라는 계층형 기억 구조로, 입력을 정규화·청킹·점수화·주제 요약·전역 다이제스트까지 연결합니다. 문서상 이 기억 계층은 단순 벡터 검색이 아니라 요약 트리와 주제 트리를 가진 로컬 마크다운 기반 위키 구조입니다. 즉 제품 본질은 “질문에 답하는 모델”보다 “질문 전에 이미 기억을 쌓아두는 시스템”에 가깝습니다. 개인 AI 시장의 경쟁축이 모델 품질에서 메모리 설계와 통합 허브로 이동하고 있음을 잘 보여 줍니다.
**기술적 배경**: 벡터스토어 단층 구조만으로는 “오늘 무슨 일이 있었는가” 같은 시간·주제·전역 요약 질문을 잘 다루기 어렵습니다. OpenHuman은 이를 요약 트리, 주제 트리, 글로벌 다이제스트로 푸는 방향을 취합니다.
**영향 분석**: 생산성 도구 시장에서는 기억 구조가 곧 해자가 될 가능성이 큽니다. 다만 로컬 우선을 내세우더라도 실제 운영 경로에 호스팅 의존이 얼마나 남는지는 계속 검증해야 합니다.
**Master 액션 포인트**:
- RAG보다 먼저 “요약 트리 + 주제 트리 + 주기적 인제스트” 조합을 내부 기억 계층 설계의 기본 후보로 두는 편이 좋습니다.
- 장기 기억 제품은 모델 교체보다 저장 구조와 프라이버시 경계를 먼저 설계해야 합니다.
- 원문: [OpenHuman GitHub](https://github.com/tinyhumansai/openhuman)
- 교차확인: [Memory Trees](https://tinyhumans.gitbook.io/openhuman/features/obsidian-wiki/memory-tree)

### 3. React Doctor — AI가 생성한 React 코드를 정적 분석으로 검증하는 진단 도구 (13pts)
**요약**: React Doctor는 AI가 생성한 React 코드의 품질을 결정론적으로 점검하는 전용 감사 도구입니다. 공식 문서상 검사 범위는 상태·이펙트, 성능, 아키텍처, 보안, 접근성, 프레임워크별 패턴까지 넓습니다. 중요한 점은 이 도구가 기존 ESLint 대체제가 아니라, PR·CI·변경분 검토에 덧대는 React 전용 검증 레이어라는 점입니다. 즉 “AI가 코드를 더 빨리 쓴다”는 흐름의 반대편에서 “누가 구조적 품질을 검사할 것인가”를 제품화한 사례입니다. React 생태계에서 검증 하니스의 상품성이 커지고 있다는 증거로 읽을 만합니다.
**기술적 배경**: 일반 린트는 문법과 규칙 위반을 잘 잡지만, 상태 관리 흐름이나 React 특유의 렌더링 안티패턴은 충분히 잡지 못하는 경우가 많습니다. React Doctor는 이 틈새를 노린 결정론적 스캐너 포지션입니다.
**영향 분석**: React 팀은 AI 생성 코드가 늘수록 생성량보다 후단 검증 도구에 더 많이 투자하게 될 가능성이 큽니다. 특히 대형 프런트엔드 코드베이스에서는 리뷰 부채를 줄이는 보조선이 될 수 있습니다.
**Master 액션 포인트**:
- React·Next 계열 작업에는 테스트 이전에 “변경분 중심 구조 검사” 단계를 CI에 따로 거는 편이 유효합니다.
- 에이전트가 생성한 프런트엔드 코드는 범용 lint보다 프레임워크 전용 진단을 먼저 붙이는 것이 안전합니다.
- 원문: [React Doctor GitHub](https://github.com/millionco/react-doctor)
- 교차확인: [React Doctor Docs](https://www.react.doctor/docs)

### 4. Codex, 활용 사례 모음 대폭 확장 (71pts)
→ 원문: [Codex use cases](https://developers.openai.com/codex/use-cases)
→ 교차확인: [Introducing Codex](https://openai.com/index/introducing-codex/)
**요약**: OpenAI는 Codex 활용 사례 허브를 기존 소수 예시에서 대규모 카탈로그로 확장하며, 제품 포지션을 코드 보조에서 업무 단위 에이전트로 옮기고 있습니다. 소개 문서와 제품 발표를 함께 보면 핵심은 코드 생성이 아니라 격리된 샌드박스, 병렬 작업, 로그·테스트 근거 제시, 목표 단위 작업 실행입니다. 제3자 해설도 이 변화를 “12개 예시에서 52개 실무 템플릿으로 커진 전략적 전환”으로 읽습니다. 즉 Codex는 더 잘 쓰는 코더가 아니라, 더 잘 분해된 backlog를 소화하는 엔지니어링 워크 엔진이 되려 합니다. 이 변화는 앞으로 경쟁 포인트가 모델 점수보다 업무 템플릿·환경 계약·검증 루프가 될 것임을 보여 줍니다.
**기술적 배경**: 제품 설명에 따르면 각 작업은 독립된 환경에서 돌아가고, 파일 편집·명령 실행·테스트 반복을 수행하며, AGENTS.md 같은 저장소 규약도 읽습니다. 이는 채팅형 보조도구보다 작업 운영체제에 가까운 구조입니다.
**영향 분석**: 조직은 Codex를 IDE 보조보다 반복 업무 자동화 계층으로 재평가할 가능성이 큽니다. 동시에 backlog를 잘게 정의하고 종료 조건을 명확히 적는 팀이 더 큰 효과를 보게 됩니다.
**Master 액션 포인트**:
- OpenClaw 작업도 단발 프롬프트보다 `목표 / 종료 조건 / 검증 명령 / 산출물 경로`를 먼저 적는 Codex식 계약으로 통일하십시오.
- iOS·게임·콘텐츠 파이프라인 모두 “무슨 모델인가”보다 “무슨 작업 템플릿으로 맡길 것인가”를 먼저 자산화하는 편이 낫습니다.

### 5. AI를 사용해 더 나은 코드를 더 천천히 작성하기 (38pts)
→ 원문: [Using AI to write better code more slowly](https://nolanlawson.com/2026/05/25/using-ai-to-write-better-code-more-slowly/)
→ 교차확인: [Project Glasswing: An initial update](https://www.anthropic.com/research/glasswing-initial-update)
**요약**: Nolan Lawson은 AI 코딩의 목적을 “더 빨리 더 많이 생성하기”가 아니라 “더 느리더라도 더 깊게 검증하기”로 재정의합니다. 글의 핵심은 Claude, Codex, Bugbot 같은 서로 다른 모델을 병렬 리뷰어처럼 써서 잠복 버그와 설계 결함을 드러내는 워크플로입니다. Anthropic의 Glasswing 업데이트도 취약점 탐지 속도가 폭증했지만 실제 병목은 검증·공개·패치라고 밝히며 이 문제의식을 뒷받침합니다. 결국 중요한 것은 생성량이 아니라 실패 모드를 얼마나 빨리 발견하고 폐기 결정을 내릴 수 있느냐입니다. 바이브 코딩 열풍에 대한 가장 실무적인 교정선으로 봐도 좋습니다.
**기술적 배경**: 다중 모델 리뷰는 단일 모델의 환각과 과신을 줄이는 현실적 방법입니다. 생성 모델을 작성기보다 감사기와 반례 탐색기로 쓰는 패턴이 점점 강해지고 있습니다.
**영향 분석**: 팀이 AI 도입 KPI를 코드량이나 PR 처리량으로 잡으면 오히려 품질 사고가 커질 수 있습니다. 반대로 테스트 보강, 회귀 탐지, 설계 오류 조기 발견에 AI를 배치하면 장기 유지보수 비용을 낮출 수 있습니다.
**Master 액션 포인트**:
- 에이전트 코딩은 `생성 → 다중 리뷰 → 테스트/로그 확인 → 수동 승인` 구조를 기본값으로 두십시오.
- 대형 자동 수정은 병합보다 “문제 탐지와 PR 폐기” 권한부터 설계하는 편이 안전합니다.

### 6. Figma 디자인 에이전트 공개 (12pts)

### 7. LLM 평가의 맹점: 우리는 왜 '행동'이 아니라 '지식'만 보는가? (12pts)

**기술적 배경**: 기존 벤치마크는 최종 성공 여부에 치우쳐 있고, 실제 에이전트 운용에서 중요한 중간 의사결정과 복구 행동을 잘 기록하지 못합니다. 그래서 로그 기반 행동 평가가 새 층위로 부상하고 있습니다.
**영향 분석**: 모델 선정 기준이 리더보드에서 역할 적합성으로 이동할 가능성이 큽니다. 내부 태스크 로그를 쌓는 팀이 장기적으로 더 강한 모델 평가 자산을 가지게 됩니다.
**Master 액션 포인트**:
- 내부 에이전트 평가표에 pass/fail 외에 탐색 단계 수, 재시도 품질, 회복 전략을 남기십시오.

### 8. React를 좋아하는 사람이 실제로 있긴 한가요? (11pts)

**영향 분석**: 프런트엔드 팀은 “다들 React를 쓰니까”라는 이유만으로 선택하기 더 어려워질 수 있습니다. 제품 제약과 조직 역량에 맞는지 먼저 검토하는 문화가 중요해집니다.
**Master 액션 포인트**:
