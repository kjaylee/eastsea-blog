### 1. ponytail - AI 에이전트를 가장 게으른 시니어 개발자처럼 생각하게 만들기 (55pts)
**[ponytail - AI 에이전트를 가장 게으른 시니어 개발자처럼 생각하게 만들기](https://github.com/DietrichGebert/ponytail)**
**요약**: Ponytail은 AI 코딩 에이전트에게 “무엇을 더 만들까”가 아니라 “무엇을 지울까”를 먼저 묻게 만드는 규칙 기반 스킬입니다. GitHub 원문은 실제 Claude Code 세션 12개 작업, Haiku 4.5 기준으로 평균 코드량 54% 감소, 비용 20% 절감, 속도 27% 개선을 주장합니다. 핵심은 모델을 바꾸지 않고 세션 시작 규율과 리뷰 질문을 바꿔 과잉 설계와 불필요한 추상화를 줄였다는 점입니다. 단순한 한 줄 규칙처럼 보이지만, 실제로는 diff 검토·삭제 목록·지연된 개선 항목 수확 같은 실행 절차가 포함되어 있습니다. 한국어 해설도 이를 “최소한의 코드로 움직이게 하는 스킬”로 정리하고 있어, 커뮤니티 수용도 빠른 편입니다. 즉 오늘의 관심은 더 강한 모델이 아니라, **에이전트가 덜 쓰고도 더 안전하게 끝내게 만드는 운영 프롬프트 계층**에 있습니다.
**기술적 배경**: 2026년 코딩 에이전트의 주된 문제는 생성 능력 부족이 아니라 과잉 구현과 컨텍스트 드리프트입니다. Ponytail은 YAGNI를 에이전트 운영 규칙으로 고정해, UI 컴포넌트 추가·추상화 도입·라이브러리 설치 같은 흔한 과잉 반응을 줄입니다. 범용 모델 성능 향상과 달리, 이런 스킬은 즉시 재현 가능하고 비용 대비 효과가 큽니다.
**영향 분석**: 개발자에게는 “좋은 프롬프트”보다 “반복 가능한 세션 규율”이 생산성 레버리지라는 점을 보여줍니다. 스타트업은 같은 모델로도 개발 단가를 낮출 수 있고, 인디 빌더는 작은 프로젝트에서 과설계로 새는 시간을 줄일 수 있습니다. 특히 릴리스 속도가 중요한 우리 같은 자동화 환경에서는 아주 작은 diff를 유지하는 문화가 장애 복구 속도까지 올립니다.
**Master 액션 포인트**: OpenClaw 코드 작업 프롬프트에 `삭제 후보 먼저 제시 → 네이티브/기존 기능 우선 → 새 의존성 도입 사유 필요` 규칙을 공통 가드레일로 넣으십시오. eastsea와 게임 파이프라인에는 “추상화 추가 전 중복 2회까지 허용” 원칙을 문서화하는 편이 좋습니다.

### 2. 구글을 AI로 해킹해서 7억원 벌기 (30pts)
**[구글을 AI로 해킹해서 7억원 벌기](https://brutecat.com/articles/hacking-google-with-ai/)**
**요약**: 보안 연구자 Brutecat은 Google의 discovery document와 대량 API 키를 기반으로, AI가 API 표면을 자동으로 퍼징하도록 만들어 3개월간 약 50만 달러의 버그 바운티를 획득했다고 설명합니다. 글의 요점은 AI가 취약점을 “발견했다”기보다, 사람이 다 보기 어려운 방대한 API 조합을 **반복·탐색·우선순위화**하는 데 강력한 보조도구가 되었다는 점입니다. 특히 6만 개 이상의 Google APK를 긁어 API 키를 모으고, discovery document를 통해 내부 API 표면을 구조화한 뒤, 에이전트가 의미 있는 요청 조합을 대규모로 시도하게 했다는 흐름이 중요합니다. 이건 단발성 해킹담이 아니라, LLM이 보안 연구의 검색·조합·자동화 계층에 들어온 사례입니다. 동시에 공개 스펙과 과도한 API 노출이 얼마나 넓은 공격면을 만드는지도 다시 드러냈습니다. 즉 오늘 항목의 핵심은 AI의 마법이 아니라 **탐색 비용이 거의 0에 가까워졌을 때 생기는 보안 현실 변화**입니다.
**기술적 배경**: Swagger나 discovery document 같은 기계가 읽는 API 명세는 원래 개발 편의를 위해 존재하지만, 공격자에게도 구조화된 지도 역할을 합니다. AI 에이전트는 여기에 퍼징 전략과 실패 패턴 학습을 얹어, 사람이 일일이 조작할 때보다 훨씬 큰 표면을 빠르게 훑을 수 있습니다. 기존 수동 버그 바운티와 달리 “문서화된 대규모 표면 + 자동 반복 탐색”이 결합된 점이 차별점입니다.
**영향 분석**: 개발자와 스타트업에게는 내부 API·테스트 엔드포인트·노출된 키·과도한 discovery 문서가 곧 자동 공격의 연료가 된다는 경고입니다. 인디 빌더도 작은 프로젝트라 방심할 수 없습니다. 한 번 외부로 공개된 디버그 라우트나 넓은 권한의 키는 에이전트 기반 스캐닝에 매우 취약합니다.
**Master 액션 포인트**: OpenClaw와 eastsea 계열 서비스에서 디버그 엔드포인트, 공개 JSON 스키마, 클라이언트 번들 내 키 노출 여부를 정기 점검하는 크론을 두십시오. 에이전트 자동화 기능을 늘릴수록 `공개 표면 인벤토리` 문서를 함께 유지하는 편이 안전합니다.

### 3. Moebius: 0.2B 이미지 인페인팅 모델로 10B급 성능 달성 (3pts)
**[Moebius: 0.2B 이미지 인페인팅 모델로 10B급 성능 달성](https://hustvl.github.io/Moebius/)**
**요약**: HUST와 VIVO AI Lab이 공개한 Moebius는 0.2B 규모 경량 인페인팅 모델이 10B급 범용 산업 모델 수준 성능에 근접하거나 일부 벤치마크에서 상회한다고 주장합니다. 프로젝트 페이지와 arXiv 초록에 따르면, 핵심은 diffusion backbone을 Local-λ Mix Interaction 블록으로 재구성하고, latent 공간에서 adaptive multi-granularity distillation을 적용해 표현력 손실을 줄인 데 있습니다. 즉 무작정 작은 모델을 만드는 것이 아니라, 인페인팅에 필요한 상호작용만 남기도록 구조를 다시 설계한 접근입니다. 이 덕분에 계산 비용과 배포 장벽을 크게 낮추면서도 고품질 복원을 유지하려는 방향이 뚜렷합니다. 오늘 항목이 주목받는 이유는 “초거대 모델이면 무조건 낫다”는 감각을 정면으로 흔들기 때문입니다. 제품화 관점에서는 모바일·온디바이스·저비용 배치 가능성이 더 커집니다.
**기술적 배경**: 이미지 인페인팅은 빈 영역만 메우는 것처럼 보여도 전역 문맥과 지역 세부 묘사를 함께 잡아야 해서 대형 모델이 유리했습니다. Moebius는 그 전역·지역 상호작용을 선형 행렬 요약과 증류 전략으로 압축해 효율을 얻습니다. 기존 대안 대비 차별점은, 단순 축소판이 아니라 **특정 과업에 맞춘 구조적 재설계**라는 점입니다.
**영향 분석**: 개발자와 스타트업은 생성 품질이 아니라 비용 대비 품질로 경쟁할 여지가 더 커집니다. 인디 빌더에게도 서버 GPU를 크게 쓰지 않고 인페인팅 기능을 붙일 수 있는 길이 열립니다. 특히 썸네일 보정, 게임 에셋 수정, 카메라 앱 후처리 같은 좁은 문제에서는 이런 특화 모델이 더 현실적입니다.
**Master 액션 포인트**: 게임·카메라 파이프라인에서 “대형 범용 모델 우선” 대신 “과업 특화 경량 모델 우선” 평가표를 만들어 보십시오. MLX/로컬 추론 자산과 연결해, 경량 비전 모델 실험 슬롯을 하나 따로 두는 것이 좋습니다.

### 5. bigset - 세상의 모든 데이터를 가지고 있다면? (16pts)
**[bigset - 세상의 모든 데이터를 가지고 있다면?](https://github.com/tinyfish-io/bigset)**

**기술적 배경**: 추상화는 처음엔 아름답지만, 요구사항 차이가 누적되면 조건 분기와 의미 붕괴를 낳습니다. AI 코딩은 이 패턴을 더 자주 유발할 수 있습니다. 기존 인간 중심 리팩터링 논의와 달리, 지금은 **잘못된 추상화의 생성 속도 자체가 빨라졌다**는 점이 다릅니다.
