### 2. AI시대, 나의 전문성을 재설계하는 법 [구글 슬라이드, 165P] (61pts)
**[2] AI시대, 나의 전문성을 재설계하는 법 [구글 슬라이드, 165P]**
**요약**: 하용호의 발표는 조직의 AX 여정을 환호→정체→신남→의구심→마지막 고비의 다섯 단계로 설명하며, 실제 병목이 모델 접근권이 아니라 검증세와 맥락 손실에서 나온다고 짚습니다. 발표에서 가장 강한 개념은 `AI J-curve Trap`과 `Verification Tax`인데, AI를 붙였다고 바로 생산성이 오르지 않고 오히려 검증 비용의 구덩이를 먼저 지난다는 주장입니다. 또 기술부채뿐 아니라 인지부채, 의도부채를 따로 분리해 “이 결과물을 왜 이렇게 만들었는지”가 조직에서 증발하는 현상을 문제로 봅니다. 사람의 메인 작업이 생산에서 검증으로 이동하며, 좋은 검증은 테스트·정량 지표·정성 루브릭의 조합으로 설계되어야 한다는 점도 중요합니다. 결론은 AI가 전문가를 대체한다기보다, 전문가의 정의를 스킬 숙련자에서 운영 책임자로 바꾸고 있다는 것입니다.

### 3. Show GN: CerberusBeacon - 필요할 때만 여는 셀프호스팅 웹 터미널 + Cloudflare 터널 게이트 (1pts)

**요약**: CerberusBeacon은 집이나 연구실의 리눅스 서버를 외부에 계속 노출하지 않고, 필요할 때만 Cloudflare Quick Tunnel로 열었다가 유휴 시 자동으로 닫는 단일 프로세스 도구입니다. GitHub README와 GeekNews 설명을 합치면 웹 터미널, 터널 제어, Telegram/Slack 봇, CLI 에이전트, 파일 브라우저, 세션 로그까지 한 묶음으로 설계되어 있습니다. 보안 면에서는 서버를 127.0.0.1에만 바인딩하고, 브라우저 로그인과 에이전트 API에 별도 TOTP 시크릿을 쓰는 점이 인상적입니다. 핵심 사용성은 ‘VPN도 포트포워딩도 없이 브라우저 하나로 잠깐 열고 닫는다’는 것입니다. 상시 공개 서비스보다 유지형 운영툴에 더 잘 맞는 구조입니다.

### 4. Paca - 인간과 AI 에이전트 협업을 위한 오픈소스 프로젝트 관리 도구 (1pts)
**[4] Paca - 인간과 AI 에이전트 협업을 위한 오픈소스 프로젝트 관리 도구**
**요약**: Paca는 Jira, Trello, ClickUp, Monday의 대체제를 표방하지만, 실제 차별점은 AI를 주변 자동화가 아니라 같은 보드 위의 스크럼 팀원으로 올려둔다는 점입니다. README 기준으로 에이전트는 스프린트에 배정되고, 백로그를 집어가며, BDD 시나리오와 시스템 설계 문서 작성에도 참여합니다. **기술적 배경**: OpenHands 기반의 격리된 에이전트 실행, Socket.IO 실시간 업데이트, WASM capability sandbox가 핵심입니다. configuration-driven 설계라 워크플로를 코드 수정 없이 조정할 수 있다는 점도 실무적입니다. 기존 PM 도구 대비 차별점은 자동화가 아니라 협업 엔티티 모델 자체를 바꾼 데 있습니다.
**영향 분석**: 개발자 팀에게는 AI를 티켓 생성기보다 작업 주체로 다루는 실험장이 됩니다. 스타트업에는 매력적이지만, 실제 성과는 검증 규율과 역할 정의가 따라붙을 때만 납니다.

### 5. 새로운 HTTP QUERY 메소드 (24pts)
**[5] 새로운 HTTP QUERY 메소드**
**요약**: RFC 10008의 `QUERY` 메소드는 GET의 URL 길이 한계와 POST의 애매한 의미론 사이를 메우기 위해 등장했습니다. QUERY는 요청 본문을 사용하지만 안전(safe)하고 멱등(idempotent)하므로, 복잡한 검색을 POST로 우회하던 오래된 관행을 더 정직한 계약으로 바꿀 수 있습니다. RFC 문서가 말하듯 QUERY는 캐시와 자동 재시도, 결과 리소스 부여 같은 표준 HTTP 기능과 더 잘 결합됩니다. 즉 `/search`에 POST를 날리면서도 사실상 읽기 전용이었던 수많은 API가 표준적 재정렬 기회를 얻는 셈입니다. 당장 브라우저·프록시·게이트웨이 생태계가 한꺼번에 따라오지는 않겠지만, 설계 방향성은 이미 분명합니다.
**기술적 배경**: 핵심 차별점은 “body를 가진 read”를 명시적 메소드로 분리했다는 점입니다. RFC 10008은 Accept-Query, Content-Location, Location 같은 필드와 함께 QUERY의 의미론을 규정해 GET/POST 혼용의 회색지대를 줄입니다. 대규모 필터, 분석성 질의, 검색 DSL 같은 영역에서 특히 자연스럽습니다.
**영향 분석**: 개발자에게는 검색 API를 더 정확한 계약으로 설계할 기회가 생깁니다. 스타트업과 인디 빌더는 장기적으로 분석용·대시보드용 엔드포인트를 더 안정적으로 캐시·재시도하게 만들 수 있습니다.
**Master 액션 포인트**: OpenClaw·eastsea 내부 API 중 “읽기 전용인데 POST를 쓰는 엔드포인트”를 목록화해 두십시오. 공개 전환은 생태계 지원을 본 뒤 결정하되, 내부 계약부터 QUERY 친화적으로 정리하는 편이 좋습니다.
- 원문: [The new HTTP QUERY method explained](https://kreya.app/blog/new-http-query-method-explained/)
- 교차확인: [RFC 10008: The HTTP QUERY Method](https://www.rfc-editor.org/rfc/rfc10008.html)

### 14. ponytail - AI 에이전트를 가장 게으른 시니어 개발자처럼 생각하게 만들기 (65pts)
**[14] ponytail - AI 에이전트를 가장 게으른 시니어 개발자처럼 생각하게 만들기**
**요약**: ponytail은 AI 코딩 에이전트가 새 라이브러리와 추상화를 과잉 도입하는 습관을 줄이기 위해, 최소 구현 사다리를 규칙으로 주입하는 프로젝트입니다. README가 제시한 벤치마크에 따르면 실제 에이전트 세션 기준 평균 54% 적은 코드, 20% 낮은 비용, 27% 빠른 시간, 그리고 안전 장치 100% 유지라는 수치를 주장합니다. 사다리의 핵심은 “이게 정말 필요한가 → 이미 코드베이스에 있는가 → 표준 라이브러리로 되는가 → 네이티브 플랫폼 기능으로 되는가 → 설치된 의존성으로 되는가 → 한 줄로 끝나는가” 순서입니다. 즉 소설처럼 길고 똑똑해 보이는 코드를 막고, 게으른 시니어의 감각을 에이전트에 이식하려는 시도입니다. 과장된 유머 포장과 별개로, 실제로는 에이전트 과설계 문제를 겨냥한 꽤 실전적인 규율 세트입니다.
**기술적 배경**: 단순 “짧게 써라” 프롬프트가 아니라 lifecycle hooks와 규칙 주입을 조합해 항상 적용되도록 만든 점이 중요합니다. 또 안전·검증은 줄이지 않고 구조적 과잉만 줄이려 한다는 메시지를 분명히 합니다. 요즘 코딩 에이전트의 가장 흔한 실패 패턴을 정면으로 건드립니다.
**영향 분석**: 개발자 팀은 작은 기능 하나에 라이브러리와 컴포넌트를 과잉 도입하는 문제를 줄일 수 있습니다. 인디 빌더에게는 유지보수성과 추론 비용을 함께 낮출 수 있는 실용적 힌트가 됩니다.
**Master 액션 포인트**: OpenClaw 코딩 지시에도 “기존 자산 재사용 → 표준 기능 우선 → 최소 구현” 사다리를 더 노골적으로 넣는 편이 좋습니다. 특히 게임 프로토타입과 도구 페이지에서는 과설계 방지 규율이 직접적인 속도 차이를 만들 수 있습니다.

### 15. Code as Agent Harness — 코드를 에이전트의 실행 기반으로 보는 102페이지 서베이 (18pts)
**[15] Code as Agent Harness — 코드를 에이전트의 실행 기반으로 보는 102페이지 서베이**
**요약**: `Code as Agent Harness`는 코드를 reasoning substrate, action interface, environment model, verification layer로 동시에 다루는 관점을 제시합니다. 프로젝트 웹페이지와 arXiv 초록은 이를 인터페이스 계층, 메커니즘 계층, 스케일링 계층의 세 층으로 정리하며, 계획·메모리·도구 사용·테스트·멀티에이전트 협업을 하나의 하네스 문제로 묶습니다. 핵심 메시지는 좋은 프롬프트보다 좋은 하네스가 장기 실행을 버티게 만든다는 것입니다. 실패도 dead end가 아니라 repair signal로 바뀌고, 저장소·상태·테스트가 곧 에이전트의 외부 기억이 됩니다. 그래서 이 서베이는 모델 벤치마크 중심 담론에서 운영 공학 중심 담론으로 넘어가는 전환점을 꽤 또렷하게 보여줍니다.
**기술적 배경**: 이 논문은 단일 에이전트보다 공유 상태와 검증 가능한 협업이 중요해지는 흐름을 잘 요약합니다. 실행 피드백, 지속 메모리, 테스트, 권한, 인간 감독이 모두 코드 층에 묶여야 한다는 관점은 오늘의 실전 에이전트 운영과 정확히 맞닿습니다. 학술 서베이지만 현장 체크리스트로 번역하기 좋습니다.
**영향 분석**: 개발자는 모델 교체보다 실패를 증거로 바꾸는 런타임에 더 많은 투자 이유를 얻습니다. 스타트업과 인디 빌더도 채팅 UI보다 상태 지속성과 검증 가능한 워크플로가 더 긴 경쟁력을 만들 가능성이 큽니다.
