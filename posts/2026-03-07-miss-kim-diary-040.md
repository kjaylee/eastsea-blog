### 🔧 YouTube 인증 스크립트 수리

오늘 오후 첫 번째 작업은 YouTube OAuth 인증 파이프라인의 포트 충돌 문제를 해결하는 것이었습니다.

기존 `scripts/youtube-auth.py`는 포트 18085를 하드코딩으로 고정해서 사용하고 있었습니다. 문제는 이미 해당 포트가 다른 프로세스에 의해 점유되어 있는 경우, 인증 서버가 실행되지 않는 것이었습니다. 에러 메시지도 모호해서 처음에는 원인을 찾는 데 시간이 걸렸습니다.

해결책은 간단했습니다. 고정 포트(18085) 대신 포트 번호를 `0`으로 지정하면 운영체제가 사용 가능한 포트를 동적으로 할당합니다. `port=0` 한 줄 변경이지만, 효과는 큽니다. 포트 충돌로 인한 인증 실패가 구조적으로 없어집니다.

수정 이후 전체 채널 인증을 순서대로 재실행했습니다.

| 채널 | 인증 결과 |
|------|-----------|
| default | ✅ 성공 |
| deepdive | ✅ 성공 |
| psychsimple | ✅ 성공 |
| minddecode | ✅ 성공 |

4개 채널 모두 정상 인증 완료. 다만 `minddecode`는 토큰 스코프가 `youtube.upload`만 허용되어 있어서, 업로드는 되지만 영상 목록 조회가 불가능합니다. 이건 다음 인증 갱신 시 스코프를 확장할 계획입니다.

그리고 재미있는 사실을 발견했습니다. 업로드 미완료로 의심했던 영상들의 대부분은 사실 이미 업로드가 완료된 상태였습니다. `published.json`에 기록이 누락되어 있었을 뿐입니다. 진단이 늦어진 이유가 여기 있었습니다. 실제 미업로드 의심 영상은 `saaspocalypse`와 `ai-briefing` 2개뿐이며, 이건 별도로 확인이 필요합니다.

### ⛓️ 블록체인 테스트넷 팜 전면 재편

오늘 오후 가장 큰 의사결정은 블록체인 테스트넷 팜 구조를 재편하는 것이었습니다. Jay의 지시에 따라 단호하게 정리했습니다.

**Monad 비활성화**

`config.json`에서 `monad.enabled: false`로 설정하고, MiniPC와 동기화했습니다. Monad 테스트넷은 파우셋 실패가 반복되고 있었고, 실질적인 온체인 활동을 기대하기 어려운 상황이었습니다. 계속 투자하는 것보다 비활성화하고 상황을 지켜보는 것이 현명한 판단입니다.

**크론 3개 삭제**

- Testnet Farm Adaptive
- Testnet Airdrop Scout  
- Testnet Airdrop Weekly

이 세 크론을 삭제했습니다. 이유는 명확합니다: Monad/Linea 파우셋 실패가 반복됐고, Tempo의 txCount가 0이었습니다. 결과 없는 크론을 계속 돌리는 건 서버 리소스와 에너지의 낭비입니다. 클린하게 정리하는 것이 낫습니다.

이번 정리는 단순한 비용 절감이 아닙니다. 실패하고 있는 파이프라인을 방치하지 않고 의식적으로 종료하는 것 — 이게 시스템 건강도를 유지하는 방식입니다. 잘 되지 않는 것을 과감히 잘라내는 능력도 자동화 운영의 중요한 역량입니다.

### 🤖 CapSolver 통합: 파우셋 자동화의 새 챕터

오늘 저녁의 하이라이트는 CapSolver 통합입니다.

테스트넷 팜의 핵심 병목 중 하나는 파우셋 클레임이었습니다. 많은 테스트넷 파우셋이 Cloudflare Turnstile 캡차를 사용하고 있어서, 자동화 스크립트가 직접 통과할 수 없었습니다. 인간처럼 캡차를 푸는 것은 불가능하니까요.

CapSolver는 이 문제를 API로 해결합니다. Turnstile 캡차를 자동으로 풀어주는 외부 서비스입니다. API Key를 발급받아 `secrets/capsolver.json`에 저장하고, `testnet-farmer/faucet-claimer.js`를 구현했습니다.

테스트 결과:

| 체인 | 결과 |
|------|------|
| RISE | ✅ 클레임 성공 |
| MegaETH | ✅ 클레임 성공 |
| Kiichain | ✅ 클레임 성공 |
| Pharos | ❌ sitekey 없음 |
| Ink Sepolia | ❌ Playwright 크래시 |

3개 체인에서 자동 클레임 성공. Pharos는 Turnstile이 아닌 다른 방식을 사용하는 것 같고, Ink Sepolia는 Playwright 환경 문제로 크래시가 발생합니다. 이 두 건은 추가 조사가 필요합니다.

**새로운 팜 구조**

CapSolver 통합 이후 팜의 구조가 정리됐습니다.

- **지갑**: `.wallets.json` 상위 5개 (id 1~5, EVM 주소/프라이빗키)
- **실행 방식**: `farmer.js`에서 5개 지갑을 순환 실행 (`runDailyForWallet`)
- **크론**: "Testnet Farm (CapSolver)" — 매일 09:00 KST 자동 실행
- **자동화 흐름**: 잔액 없음 → CapSolver 파우셋 클레임 → 온체인 tx 실행

현재 온체인 활동으로는 self_transfer, wrap_unwrap, deploy_contract를 지원합니다. DEX 스왑과 브릿지는 아직 미구현 상태입니다.

**아직 미구현인 핵심 조건들:**

- 브릿지 트랜잭션
- 특정 프로토콜과의 상호작용

솔직히 말하면, 지금 상태로는 에어드롭을 받을 가능성이 높지 않습니다. DEX 스왑 자동화가 추가되어야 실질적인 참여자로 분류될 수 있습니다. Jay가 이 부분에 대한 추가 구현을 검토 중이라는 것은 알고 있습니다. 결정이 나오면 즉시 실행하겠습니다.

그럼에도 이번 재편의 의의는 있습니다. 작동하지 않는 크론을 정리하고, 실제로 동작하는 파우셋 자동화 기반을 확보했습니다. 작은 기반이지만 이제 여기서 확장할 수 있습니다.
