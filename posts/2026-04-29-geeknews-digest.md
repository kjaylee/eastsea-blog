### 1. 하네스 엔지니어링: 모델보다 중요한 작업 환경 설계의 시대 (36pts)
→ 원문: [Agent Harness Engineering](https://addyosmani.com/blog/agent-harness-engineering/)
→ 교차확인: [Anthropic Harness Design](https://www.anthropic.com/engineering/harness-design-long-running-apps)
**요약**: Addy Osmani는 이제 코딩 에이전트의 성패가 모델보다 하네스, 즉 프롬프트, 툴, 컨텍스트 정책, 훅, 샌드박스, 피드백 루프, 복구 경로에 더 크게 좌우된다고 정리합니다. 글의 핵심은 에이전트 실패를 일회성 실수로 넘기지 말고, 다음부터는 같은 실패가 반복되지 않도록 규칙과 검증으로 고정하라는 것입니다. Anthropic도 장기 실행 앱 개발 하네스에서 planner, generator, evaluator를 분리하고, 컨텍스트가 길어지면 compaction보다 reset + handoff artifact가 더 강력하다고 설명합니다. 결국 에이전트 엔지니어링의 본체는 모델 선택표가 아니라, 실수에서 규칙을 추출하는 운영체제에 가깝습니다. 같은 모델이라도 하네스가 달라지면 성능 체감이 크게 달라진다는 점이 오늘 가장 중요한 메시지였습니다.
**기술적 배경**: 최근 1년 동안 커뮤니티는 모델 IQ 비교에 몰두했지만, 실제 현장에서는 AGENTS.md, subagent orchestration, lint back-pressure, approval gate 같은 구조물이 성능과 안정성을 더 크게 결정하기 시작했습니다. 특히 긴 작업에서 context anxiety와 자기평가 편향이 반복적으로 드러나면서, 단일 에이전트보다 역할 분리형 하네스가 주류 패턴으로 굳어지고 있습니다.
**영향 분석**: 개발자는 더 좋은 모델을 찾는 시간보다 실패 이력을 규칙화하는 시간이 더 큰 수익을 낼 수 있습니다. 스타트업과 인디 빌더에게도 에이전트 도입의 핵심 비용은 API 요금이 아니라, 검증 훅과 승인 경계 설계가 될 가능성이 큽니다.
**Master 액션 포인트**: OpenClaw의 실패 사례를 `실패 유형 → 규칙/훅/테스트` 형태로 축적하는 하네스 레저를 만드십시오. eastsea에는 “에이전트 성능의 본체는 모델이 아니라 하네스”라는 해설을 빠르게 발행할 가치가 큽니다.
- 원문: [Addy Osmani 글](https://addyosmani.com/blog/agent-harness-engineering/)
- 교차확인: [Anthropic 장기 실행 하네스 설계](https://www.anthropic.com/engineering/harness-design-long-running-apps)

**요약**: getdesign.md는 여러 서비스의 DESIGN.md 예시를 한데 모아, 코딩 에이전트가 UI 스타일을 복제하거나 참조하기 쉬운 형태로 노출하는 컬렉션입니다. **영향 분석**: 개발자는 앞으로 UI 품질을 높이려면 코드보다 먼저 설계 문서 포맷을 손봐야 합니다. 인디 빌더는 적은 인원으로도 스타일 일관성을 더 오래 유지하고, 랜딩 페이지와 제품 UI를 더 빠르게 변형할 수 있습니다.
**Master 액션 포인트**: eastsea, OpenClaw 웹 표면, 게임 랜딩 자산에 공통으로 적용할 `DESIGN.md` 유사 단일 파일을 만드십시오. 디자인 변경 내역을 diff 가능한 자산으로 남기면 에이전트 수정 품질이 확실히 올라갑니다.

### 14. OpenAI Realtime API 기반 음성 제어 React 컴포넌트 (1pts)
**요약**: OpenAI의 realtime-voice-component 저장소는 브라우저와 React 환경에서 음성 기반 UI 제어를 구현하기 위한 레퍼런스 패키지입니다. README는 이것이 범용 오케스트레이션 프레임워크가 아니라, 앱이 소유한 좁은 툴과 명시적 상태 변경을 전제로 한 UI 패턴이라고 분명히 밝힙니다. controller, widget, ghost cursor 같은 컴포넌트를 제공하지만, 프로덕션용 장기 지원 UI 킷은 아니라고 선을 긋습니다. 동시에 Realtime API 가이드는 WebRTC, WebSocket, SIP 경로와 대화 수명주기, MCP, server-side control까지 주변 문서를 잘 정리해 두고 있습니다. 즉 이 저장소는 음성 에이전트 제품을 바로 완성해 주는 것이 아니라, “음성으로 앱을 안전하게 움직이는 표준 패턴”을 예시하는 쪽에 가깝습니다.
**기술적 배경**: 음성 UI에서 가장 위험한 부분은 자유형 브라우저 자동화가 아니라, 상태 소유권과 툴 경계를 흐리는 설계입니다. 이 컴포넌트는 앱 소유 툴, 좁은 액션, UI가 최종 상태를 책임진다는 원칙을 강하게 밀어 넣습니다.
**영향 분석**: 개발자는 음성 기능을 붙일 때 챗봇보다 “툴이 제한된 UI 컨트롤러” 모델을 더 많이 채택하게 될 수 있습니다. 인디 빌더에게도 음성 기능의 제품화 장벽이 낮아지지만, 동시에 세션 설계와 승인 흐름이 더 중요해집니다.
**Master 액션 포인트**: OpenClaw의 음성 실험은 자유형 에이전트보다, 앱 소유 narrow tool 패턴으로 시작하십시오. गेम/도구 UI에서도 voice launcher와 명시적 상태 피드백을 결합하는 설계가 더 안전합니다.

### 15. Obscura - 오픈소스 헤드리스 브라우저 (30pts)
**요약**: Obscura는 Rust로 작성된 경량 헤드리스 브라우저 엔진으로, 웹 스크래핑과 AI 에이전트 자동화에 특화되어 있습니다. README는 V8 기반 자바스크립트 실행, Chrome DevTools Protocol 호환, Puppeteer/Playwright 드롭인 호환, 30MB 메모리 사용, 내장 스텔스 기능 같은 공격적인 포지셔닝을 내세웁니다. 핵심 메시지는 분명합니다. “헤드리스 브라우저를 쓰려면 Chrome 전체를 항상 띄워야 한다”는 전제를 깨 보겠다는 것입니다. 다만 속도와 메모리 수치는 프로젝트 측 self-report이므로, 실전에서는 직접 검증이 필요합니다. 그래도 에이전트 자동화가 늘어날수록 브라우저 런타임을 더 가볍게 만들려는 시도는 계속 주목받을 가능성이 큽니다.
**기술적 배경**: 기존 브라우저 자동화는 무겁고 탐지 표면이 넓으며 컨테이너 운영 비용이 큽니다. Obscura는 엔진 자체를 재설계해 비용, 속도, 배포 단순성을 한 번에 개선하려는 방향입니다.
**영향 분석**: 개발자는 대량 스크래핑과 브라우저 자동화를 더 작은 인프라 비용으로 돌릴 수 있을 가능성을 봅니다. 스타트업과 인디 빌더에게는 “브라우저 자동화 = 무거운 Chrome”이라는 고정관념을 다시 생각하게 하는 카드입니다.
**Master 액션 포인트**: OpenClaw의 대량 웹 작업 경로에서 Chrome 계열과 Obscura류 런타임의 비용·성공률·탐지율을 벤치마크하는 작은 실험을 설계해 보십시오. eastsea에는 “에이전트 브라우저는 왜 더 가벼운 런타임을 요구하는가”라는 글감으로 좋습니다.
