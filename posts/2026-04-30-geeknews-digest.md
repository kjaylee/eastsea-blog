**영향 분석**: 개발자는 실험용과 운영용 모델을 더 쉽게 분리하고, 로컬 모델과 상용 모델을 같은 인터페이스로 다룰 수 있습니다. 다만 인증과 권한 경계를 허술하게 두면 게이트웨이가 새 공격면이 될 수 있습니다.
**Master 액션 포인트**: OpenClaw의 모델 라우팅과 비용 관리를 별도 게이트웨이 계층으로 뺄지 검토할 가치가 있습니다. 다만 기본 보호가 약한 구성은 즉시 차단하고, master key 강제 같은 규율을 먼저 붙여야 합니다.

→ 교차확인: [Anthropic Harness Design](https://www.anthropic.com/engineering/harness-design-long-running-apps)
**요약**: Addy Osmani는 코딩 에이전트 성능의 본체가 모델보다 하네스, 즉 프롬프트, 승인, 도구, 훅, 검증, 컨텍스트 정책이라고 정리합니다. Anthropic의 장기 실행 하네스 글도 같은 결론에 도달하며, planner, generator, evaluator 분리와 context reset + handoff artifact가 compaction보다 강한 경우를 상세히 설명합니다. 핵심은 에이전트 실패를 “이번 한 번의 실수”로 넘기지 말고, 다음부터는 같은 실패가 불가능하도록 규율과 구조를 만드는 데 있습니다. 특히 self-evaluation의 관대함과 context anxiety를 분리된 평가자와 강제 리셋으로 다루는 대목은 매우 실전적입니다. 오늘 리스트 전체를 관통하는 메타 주제이자, Master의 현재 작업 방식이 옳은 방향이라는 외부 검증이기도 합니다.
**기술적 배경**: 프론티어 모델이 평준화되면서 차별화 지점이 모델 IQ에서 운영 구조로 이동했습니다. 긴 작업일수록 역할 분리, 테스트 게이트, 명시적 handoff, 좁은 권한 설계가 성능과 안정성을 동시에 끌어올립니다.
**영향 분석**: 개발자는 더 좋은 모델을 찾는 시간보다 실패 로그를 규칙으로 흡수하는 시간이 더 큰 ROI를 낼 수 있습니다. 스타트업도 에이전트 기능을 붙이려면 먼저 승인 경계와 자동 검증 루프를 제품에 내장해야 합니다.
**Master 액션 포인트**: OpenClaw에서 실패 사례를 `실패 유형 → 방지 규칙 → 테스트`로 축적하는 하네스 레저를 만드십시오. eastsea에는 “에이전트 성능은 모델이 아니라 하네스의 품질”이라는 논지로 발행 가치가 큽니다.

**영향 분석**: 개발자는 UI 품질을 높이려면 코드보다 먼저 설계 문서 포맷을 정비해야 합니다. 인디 빌더에게는 일관된 브랜드 표면을 유지하면서도 에이전트에게 수정 권한을 더 많이 줄 수 있게 됩니다.
**Master 액션 포인트**: eastsea, OpenClaw 웹 인터페이스, 게임 랜딩 페이지에 공통으로 적용할 단일 디자인 스펙 파일을 만드십시오. 디자인 변경을 diff 가능한 자산으로 남기면 에이전트 수정 품질이 눈에 띄게 올라갑니다.

### 14. Show GN: Fairy - 개발자 프로젝트와 오픈소스를 후원하는 서비스 (40pts)

### 15. PgQue – Bloat 없는 Postgres 큐 (8pts)
- 원문: [NikolayS/pgque](https://github.com/NikolayS/pgque)
- 교차확인: [Hacker News 토론](https://news.ycombinator.com/item?id=47817349)
**요약**: PgQue는 Postgres 안에서 동작하는 이벤트 큐를 순수 SQL과 PL/pgSQL만으로 구현한 프로젝트입니다. README는 흔한 SKIP LOCKED 기반 큐가 결국 dead tuple, VACUUM 압박, 성능 저하로 무너진다고 지적하며, snapshot-based batching과 TRUNCATE rotation으로 그 구조적 문제를 피하겠다고 설명합니다. 설계는 Skype 시절의 PgQ를 현대적인 관리형 Postgres 환경에 되살리는 쪽에 가깝고, event log와 fan-out 소비자 모델을 전면에 둡니다. 대가로 기본 지연은 대략 1~2초 수준이며, 초저지연 작업 큐 용도에는 맞지 않는다고 솔직히 인정합니다. 즉 범용 job queue가 아니라, **오래 버티는 Postgres 내부 이벤트 스트림**에 더 가깝습니다.
**기술적 배경**: 많은 팀이 인프라를 줄이려고 Postgres 큐를 택하지만, 부하가 오르면 bloat 문제가 반복적으로 터집니다. PgQue는 이 역사를 정면으로 겨냥하면서, 관리형 Postgres 친화성과 운영 단순성을 동시에 노립니다.
**영향 분석**: 개발자는 별도 메시지 브로커 도입 전, 이벤트 로그 성격의 워크로드를 Postgres 안에서 더 단단하게 처리할 수 있는 선택지를 얻게 됩니다. 인디 빌더에게는 인프라 복잡성을 줄이는 꽤 매력적인 옵션입니다.
**Master 액션 포인트**: 실시간성보다 내구성이 중요한 워크로드가 있다면 PgQue류 설계를 검토하십시오. 다만 OpenClaw의 즉응성이 필요한 경로와는 용도를 명확히 분리해야 합니다.
