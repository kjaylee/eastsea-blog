### 1. 대부분의 스타트업은 번(burn) 문제가 아니라 의사결정 문제를 안고 있다 (4pts)

### 3. auth.md — 에이전트가 사용자를 대신해 가입시키기 위한 오픈 프로토콜 (3pts)
**요약**: WorkOS의 auth.md는 에이전트가 서비스에 사용자를 대신해 등록하고 인증받는 절차를 표준화하려는 오픈 프로토콜입니다. 서비스는 도메인 루트에 auth.md 파일을 두고, 에이전트는 이를 읽어 어떤 등록 흐름과 권한 범위가 지원되는지 파악합니다. 레퍼런스 구현은 세 역할—사용자를 대신해 행동하는 에이전트, ID-JAG를 발급하는 에이전트 프로바이더, 그리고 이를 수용해 액세스 토큰을 발급하는 서비스—로 나뉩니다. ID-JAG가 불가능한 경우에는 RFC 8628 스타일의 클레임 절차로 사용자 확인을 거친 뒤 자격증명을 발급하는 대체 흐름도 제공합니다. 즉, 기존 “회원가입 폼 + 브라우저 수동 입력”을 에이전트 친화적 등록 프로토콜로 바꾸려는 시도입니다. 아직 초기 단계지만, 이미 Cloudflare·Resend·Firecrawl 등 예시 도입처가 보이고 GitHub 레퍼런스 저장소도 함께 공개되어 있습니다.
**기술적 배경**: 에이전트 시대의 인증은 단순 OAuth로 끝나지 않습니다. 문제는 “이 에이전트가 누구를 대신해 왜 이 서비스에 접근하는가”를 서비스가 이해할 수 있어야 한다는 점입니다. auth.md는 기존 OAuth 메타데이터와 JWT bearer grant, JWKS, 기기 코드류 claim ceremony를 조합해 에이전트 등록에 특화한 발견·등록·교환 흐름을 제시한다는 점에서 차별화됩니다.
**영향 분석**: 개발자에게는 에이전트용 가입 경로를 별도 해킹하지 않고 표준 형태로 붙일 기회가 열립니다. 스타트업은 사용자 획득 흐름을 “사람이 직접 가입”에서 “사용자의 AI가 서비스를 붙이는” 방향으로 확장할 수 있습니다. 인디 빌더에게도 초기 폼 UX보다 에이전트 호환 메타데이터를 잘 설계하는 쪽이 미래 대비가 될 수 있습니다.
**Master 액션 포인트**: OpenClaw 외부 연동 서비스나 eastsea 툴 페이지에 장기적으로 `auth.md` 또는 유사한 agent-ready 메타데이터를 붙일지 검토하십시오. 특히 에이전트가 대신 가입·연결하는 서비스 패턴은 향후 OpenClaw 플러그인 생태계와 잘 맞습니다.

### 4. Code as Agent Harness — 코드를 에이전트의 실행 기반으로 보는 102페이지 서베이 (13pts)
**요약**: 이 서베이는 코드를 단순 생성 결과물이 아니라 에이전트의 실행 매체이자 검증 장치, 상태 저장소, 협업 인터페이스로 봐야 한다고 주장합니다. 문서가 제시하는 핵심은 에이전트가 장기 작업을 잘하려면 reasoning trace, programmable actions, DOM/API interface, simulator, tests, durable state 같은 하네스가 필요하다는 점입니다. 즉 모델이 똑똑한가보다, 실패를 다시 실행·수정·검증 가능하게 감싸는 코드 층이 더 중요하다는 이야기입니다. 서베이는 planning, memory, tool use, control, optimization이 하네스 위에서 유지될 때만 장기 실행이 버티며, 실패 역시 dead end가 아니라 repair signal이 된다고 설명합니다. 또한 여러 에이전트가 같은 저장소와 리뷰 아티팩트 안에서 협업할 수 있어야 진짜 멀티에이전트가 된다고 봅니다. 오늘 커뮤니티가 이 글에 반응한 것은 에이전트 성능 논의가 모델 벤치마크에서 런타임 설계로 이동 중이기 때문입니다.
**기술적 배경**: 기존 에이전트 논의는 툴 호출이나 프롬프트 기법에 치우친 경우가 많았습니다. 이 서베이는 테스트, 시뮬레이터, 코드 저장소, 권한, 메모리, 평가 루프를 하나의 하네스 공학 문제로 묶었다는 점이 다릅니다. 특히 OpenClaw 같은 시스템에는 이미 익숙한 이야기지만, 학술적으로 이를 구조화했다는 점에서 정리 가치가 큽니다.
**영향 분석**: 개발자는 “모델 교체”보다 “실패를 증거로 바꾸는 실행 환경” 구축에 더 투자해야 합니다. 스타트업은 에이전트를 제품 기능으로 넣을 때 채팅 UX보다 검증 가능한 워크플로와 상태 지속성에 더 예산을 써야 합니다. 인디 빌더도 코드베이스·테스트·배포 로그를 에이전트 친화적으로 정리할수록 작은 팀으로 더 큰 작업을 돌릴 수 있습니다.
**Master 액션 포인트**: OpenClaw의 spawn 지시서 표준과 검증 게이트를 이 프레임으로 재정리해 “에이전트 하네스 설계 원칙” 문서로 자산화하십시오. eastsea에는 단순 뉴스 소개보다, 서브에이전트 파이프라인이 왜 경쟁우위인지 설명하는 해설형 글이 더 어울립니다.

### 7. 새로운 HTTP QUERY 메소드 (1pts)
**요약**: Kreya 글은 최근 표준화된 RFC 10008의 HTTP QUERY 메소드를 설명합니다. QUERY는 요청 바디를 포함하되, 서버가 그 요청을 `safe`하고 `idempotent`하게 처리하도록 정의된 메소드입니다. 쉽게 말해 복잡한 검색 조건을 GET의 쿼리스트링 한계 없이 보내면서도, POST처럼 상태 변경 위험을 끌고 오지 않는 중간지대를 마련한 것입니다. GET은 URL 길이 제한과 구조 표현의 불편함이 있었고, POST는 검색 요청이어도 캐싱·재시도·프락시 의미론에서 애매했습니다. QUERY는 이런 간극을 메우기 위해 등장했고, Accept-Query 헤더로 지원 포맷을 협상할 수 있습니다. REST API가 점점 검색형·분석형 요청을 많이 다루는 흐름에서 꽤 실용적인 변화입니다.
**기술적 배경**: 중요한 차별점은 “body가 있는 read-like request”를 표준 HTTP 의미론 안으로 끌어들였다는 점입니다. 많은 API가 이미 POST /search 같은 우회 패턴을 써왔는데, QUERY는 이것을 안전하고 반복 가능한 요청으로 정식 모델링합니다. 향후 API 게이트웨이, 캐시, SDK가 이를 얼마나 빨리 수용하느냐가 관건입니다.
**영향 분석**: 개발자는 검색·필터 API 설계를 더 정교하게 만들 수 있습니다. 스타트업은 대형 필터 객체를 POST로 숨기던 관행에서 벗어나 보다 명시적인 계약을 설계할 수 있습니다. 인디 빌더에게도 분석형 백엔드가 많아질수록 URL 해킹 없이 깨끗한 프로토콜 레벨 지원을 받을 여지가 생깁니다.
**Master 액션 포인트**: OpenClaw나 eastsea 내부 API 설계 문서에서 복잡한 조회형 엔드포인트를 QUERY 후보로 분류해 두십시오. 당장 쓰지 않더라도, POST 기반 조회 API를 장기적으로 어떤 의미론으로 다룰지 정리하는 데 도움이 됩니다.

### 12. ponytail - AI 에이전트를 가장 게으른 시니어 개발자처럼 생각하게 만들기 (64pts)
**요약**: ponytail은 AI 코딩 에이전트가 불필요한 코드를 과잉 생성하지 않도록 만드는 규칙 세트이자 플러그인입니다. 저장소 설명에 따르면 실제 FastAPI+React 오픈소스 저장소에 대한 Claude Code 세션 비교에서 평균 54% 적은 코드, 20% 낮은 비용, 27% 빠른 완료 시간을 기록했고, 특정 과설계 함정에서는 94%까지 코드량을 줄였다고 합니다. 핵심은 “가장 먼저 기능 자체가 필요한가를 묻고, 이미 있는 것·표준 라이브러리·플랫폼 기본 기능·기존 의존성·한 줄짜리 해결책 순으로 사다리를 내려간다”는 점입니다. 중요한 것은 단순 코드 골프가 아니라, 검증·에러 처리·보안·접근성은 절대 깎지 않는다고 명시한 것입니다. 즉 적게 쓰되, 안전장치는 자르지 않는 실전형 미니멀리즘입니다. 지금 커뮤니티에서 크게 반응한 이유는 AI 코딩 에이전트 시대의 진짜 문제 중 하나가 “너무 많은, 너무 그럴듯한 코드”이기 때문입니다.
**기술적 배경**: 많은 프롬프트 최적화가 말투를 줄이는 수준에 머무르는 반면, ponytail은 실제 에이전트 워크플로에 삽입되는 의사결정 사다리를 제공합니다. YAGNI, 재사용 우선, 플랫폼 기본 기능 우선, 의존성 최소화 같은 오래된 시니어 감각을 AI 런타임 규칙으로 만든 점이 차별점입니다. 한국어 해설 커뮤니티에서도 이 원칙이 “게으르지만 부주의하지 않다”는 표현으로 번역되며 공감을 얻고 있습니다.
**영향 분석**: 개발자는 AI가 만든 과설계와 의존성 남발을 줄일 수 있습니다. 스타트업은 코드량 폭증에 따른 리뷰 비용과 유지보수 비용을 낮출 수 있고, 인디 빌더는 빠른 속도와 장기 관리 가능성을 동시에 챙길 수 있습니다. 특히 작은 팀일수록 한 번 들어온 불필요한 코드가 오래 남기 때문에 효과가 큽니다.
